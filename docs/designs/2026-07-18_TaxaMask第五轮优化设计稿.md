## 1. 已确认需求

| 事项 | 最终要求 | 对研究工作的意义 |
|---|---|---|
| 文件与训练可信度 | 未验证文件不得训练；不提供绕过开关 | 模型不会混入来源不明或被外部改动的数据 |
| 训练记录 | 每次训练都保存文件版本、实际生效参数、环境、状态、产物和备注关联；不区分“正式/临时训练” | 日后能解释一份模型实际用了什么，备注只负责标记重要性 |
| 推理拆分与日志 | 结果保持一致地拆分长流程，并记录 Locator、专家路由、SAM、跳过/失败原因及耗时 | 分割异常时能定位具体阶段；不是记录所谓“模型思考过程” |
| 3D 网格 | 从人工复核标签体导出 STL，以 Blender 5.0 验收；原始网格与可选展示副本分开 | 可用于外部检查、渲染和谨慎的尺度核验 |
| 高风险优先复核 | 复用现有 Local Axis 队列，只改变复核顺序并解释风险原因 | 更早看到可能出错的样本，但机器结果仍须人工确认 |
| 多标注者 | 当前分支不做 | 留给未来线上协作系统 |

### 1.1 唯一存储口径

1. 项目内的完整性版本、训练运行、网格导出状态和来源记录，只以项目 SQLite 为权威。
2. 不建设 JSON/SQLite 平级双后端，也不生成 `mesh_export_manifest.json` 网格 sidecar。
3. STL 是导出产物，不是事实来源。其相对位置、哈希、单位、轴向、来源标签版本、处理参数和完成状态全部登记到 SQLite。
4. 现有后端为了兼容而产生的 JSON manifest、report 或 JSONL 日志可以继续作为普通产物；它们不能恢复、覆盖或替代项目状态。只有被列为某次 run 正式产物的文件，才需要在 SQLite 登记角色、相对位置和哈希；普通运行日志不要求逐条写入 SQLite。
5. 项目和 run 记录不得保存本机绝对路径。项目外位置只在本机 `user_config_dir()/location_registry.sqlite` 中保存，项目库仅引用不透明位置 ID。
6. SQLite 事务不能包住外部 STL 文件写入，因此网格导出仍采用可恢复状态机：先在 SQLite 建 `pending` 记录，再原子发布已复验的 STL，最后用一个 SQLite 事务写入哈希并标记 `complete`。任何中断都只能显示为未完成、失败或待复核。

取消 JSON sidecar 的取舍是：STL 文件夹脱离原项目数据库后不再自带完整来源说明。本轮优先保证项目内记录可靠和可恢复；若以后需要制作论文附件或跨机构交换包，应单独设计一次性只读导出报告，不能把它变成第二套项目存储。

### 1.2 文件不一致时的处理

- 文件移动：用户选择新位置，完整哈希一致后更新位置引用并保留迁移记录。
- 有意修改：显式登记为新数据版本，填写备注并重新验证。
- 意外覆盖或损坏：提示来源和可用备份入口，恢复后重新校验；程序不得自动覆盖用户文件。
- 缺失且无法恢复：取消训练并保存诊断，不创建半成品权重。
- 哈希中断：保持 `incomplete`，允许重试，绝不能显示为已验证。

旧项目升级只建新表并补稳定 ID，不自动扫描或哈希文件。初始 registry 状态为 `uninitialized`；打开、保存和训练预检都不得偷偷把当前磁盘内容登记为可信基线。历史 polygon 没有来源记录时，用户完成一次 `legacy_truth_attestation` 后才可建立初始人工真值基线。

哈希策略固定为：大型原始 CT/volume 可显式采用版本化快速指纹；人工真值、标签 schema、训练配置和初始权重必须完整 SHA256。训练前不一致未解决时，训练不得启动。

### 1.3 现有人工操作不得改变

- TIF 的“通过所选 AI 结果”只处理用户明确选择的样本，可用于批量确认高置信度结果。
- Local Axis 当前仍是单项显式接受；“批量导出已接受项”不是批量通过。
- 排序、置信度或困难标记都不能自动改变真值状态。
- 只有已经提升为 `manual_truth` 或二维工作流中明确人工通过的等价记录可以训练。

## 3. 本轮需求基线

| 阶段 | 需求 | 必须达到的结果 |
|---|---|---|
| 第一阶段：现状与安全基线 | R-000 需求追踪；R-005 推理行为基线；R-016 保护现有能力 | 所有训练入口、真值门禁、推理组合和数据契约可复核，后续变化有基准 |
| 第二阶段：科研溯源 | R-001 版本化完整性；R-002 实际训练参数；R-003 损失权重配置；R-004 异常善后 | 能证明模型使用的数据、配置、环境和产物；异常有明确恢复路径 |
| 第三阶段：推理诊断 | R-006 等价拆分；R-007 分阶段日志 | 框、路由、分数、mask/polygon 在约定容差内一致，并能解释失败阶段 |
| 第四阶段：网格 | R-008 STL/SQLite 契约；R-009 导出、质量和恢复 | Blender 5.0 可正确导入；尺度/轴向可核验；中断不误报成功 |
| 第五阶段：复核效率 | R-010 高风险样本优先复核 | 风险原因可解释；原排序可恢复；真值门禁和现有人工操作无回归 |
| 第六阶段：交付 | R-011 测试与 CI 补强 | 自动测试、真实工作流验收、数据安全检查和文档同步完成 |

### 明确不进入本轮

- R-012 多标注者、在线身份、协作权限和仲裁。
- R-013 整体重写 Manager 或提前新建一套 Registry/TrainingLoop 架构。
- R-014 单独删除 `main.py` fallback。
- R-015 在没有静态检查流水线时批量增加 Mixin Protocol。
- JSON/SQLite 平级后端、网格 JSON sidecar、OBJ/PLY 交付。
- 完整的数据集发布平台、引用标识和线上共享系统；本轮只完成其可追溯产物基础。

## 4. 长任务路线

```text
第一阶段 现状与安全边界
  -> 第二阶段 文件完整性与完整训练记录
  -> 第三阶段 推理等价拆分与诊断日志
  -> 第四阶段 Blender 5.0 / STL 网格导出
```

每一阶段完成后先提交测试证据、产物位置和剩余风险，再进入下一阶段。内部的 `M0/M1` 和 `T-M1-001` 只用于执行追踪，不要求用户靠编号理解需求。

## 5. 何时需要再次确认

执行人员自行决定依赖选择、表结构细节、函数拆分、性能优化和失败重试实现，并对结果负责。只有下列情况重新请用户决定：

1. 需要改变人工真值、一键通过、未验证样本禁训等研究规则；
2. 需要不可逆地批量迁移、覆盖或删除现有用户项目和研究数据；
3. 需要默认平滑、补洞、删除结构或采用可能改变测量含义的几何处理；
4. 现有 metadata 无法可靠还原网格比例/轴向，必须改变用途承诺或要求用户补录研究信息；
5. 需要扩大到多标注者、在线协作、第二存储后端或其他本轮明确排除的范围；
6. 发现真实数据、密钥、私人路径或模型权重有泄漏风险。

普通技术问题不交给用户逐项拍板。执行人员应先修复、补测试并留档；若无法在上述需求和安全边界内解决，才按对应范围变化说明后果。
