# 标注工作台通用物种适配规划

日期：2026-05-03

状态：规划稿，待用户验收后进入代码调整

## 1. 背景

标注工作台的核心能力已经不是单纯蚂蚁专用：项目 JSON 中保存自己的 `taxonomy`，训练时 Locator 会按 `locator_scope` 重建输出通道，导出时 COCO / YOLO 也按项目 taxonomy 写类别。

但当前默认体验仍然保留较多蚂蚁研究历史：

- 默认 taxonomy 是 `Head / Mesosoma / Gaster`。
- 新项目兜底名仍是 `Formica_Seg`。
- 窗口标题仍是 `AntSleap Workbench`。
- 标注元数据字段叫 `genus`。
- COCO 导出 `supercategory` 写死为 `ant_part`。
- Multimodal JSONL schema 仍叫 `formica-multimodal-sample-v1`。
- 路径恢复逻辑包含 `Formica-Flow_output` 个人历史目录。

这些问题不一定阻止其他物种使用，但会影响开源用户的第一印象，也会污染导出的数据集元数据。

## 2. 设计目标

这次改造目标不是把蚂蚁工作流抹掉，而是把它整理成一个清楚的默认模板：

1. 保留蚂蚁模板，因为这是当前验证最充分的研究场景。
2. 新增通用项目模板，让非蚂蚁用户首次使用时不会默认继承蚂蚁部位。
3. 把“部位标签”“粗定位标签”“分类单元元数据”“导出 schema”分清楚。
4. 不破坏旧项目 JSON 的读取。
5. 不夸大跨物种自动化能力：程序提供可配置接口和模板，正式大批量使用前仍需小批量人工验收。

## 3. 用户工作流目标

面向研究者，最终应该形成这样的流程：

1. 新建项目时选择模板。
2. 如果研究对象是蚂蚁，可直接使用 Ant Taxonomy 模板。
3. 如果研究对象是其他类群，可选择 Generic Taxonomy 模板，再手工改 taxonomy。
4. 标注时右侧列表显示的是当前项目的结构标签，而不是固定蚂蚁部位。
5. 训练预检明确展示完整 taxonomy 和 locator scope。
6. 导出数据集时，不再把其他物种数据写成 `ant_part` 或 `formica-*` schema。

## 4. 模板设计

建议新增项目模板层，而不是只改默认数组。

建议文件：

- `AntSleap/core/project_templates.py`

建议内置模板：

```python
PROJECT_TEMPLATES = {
    "ant_default": {
        "display_name": "Ant morphology (validated default)",
        "taxonomy": ["Head", "Mesosoma", "Gaster"],
        "locator_scope": ["Head", "Mesosoma", "Gaster"],
        "taxon_label": "Genus",
        "category_supercategory": "ant_part",
    },
    "generic_taxonomy": {
        "display_name": "Generic taxonomy mask project",
        "taxonomy": ["Object", "Region", "Structure"],
        "locator_scope": ["Object"],
        "taxon_label": "Taxon",
        "category_supercategory": "biological_structure",
    },
}
```

设计说明：

- `ant_default` 保留当前研究基础。
- `generic_taxonomy` 给外部用户一个中性入口。
- 模板只影响新项目，不强行改旧项目。
- 用户仍可以在界面中增删 taxonomy。

## 5. 数据结构调整

### 5.1 旧字段兼容

当前项目中每张图的分类学元数据存在：

```json
"genus": "Unknown"
```

建议新结构增加：

```json
"taxon": "Unknown",
"taxon_rank": "",
"taxon_metadata": {}
```

兼容策略：

- 读取旧项目时，如果有 `genus` 且没有 `taxon`，把 `taxon` 显示为 `genus`。
- 保存时短期内可以同时保留 `genus` 和 `taxon`，避免旧逻辑断裂。
- GUI 文案先改成“Taxon / 分类单元”，底层函数可保留 `get_genus()`、`set_genus()` 作为兼容包装。

### 5.2 taxonomy 与 taxon 的区别

需要在代码和文档中保持概念清晰：

- `taxonomy`：当前项目要标注的结构类别，例如 Head、Leaf、Wing、Object。
- `taxon`：图像对应的分类单元，例如 Formica、Quercus、某个物种候选名。

这个区分对长期整合数据集很重要，否则“身体部位标签”和“生物分类名”容易混在一起。

## 6. UI 调整

### 6.1 新建项目

建议在新建项目对话流程中增加模板选择：

- Ant morphology (validated default)
- Generic taxonomy mask project

如果短期不想扩展复杂对话，可以先用一个简单下拉框；默认仍可选 Ant，但开源版本更建议默认 Generic，或者记住用户上次选择。

### 6.2 右侧元数据面板

当前右侧第一行是可编辑 `genus_combo`，但界面没有明确标签。

建议增加或改名：

- `Taxon:` / `分类单元:`
- `Structures:` / `结构标签:`

这样非蚂蚁用户会理解：

- 上方输入的是这张图对应的分类单元。
- 下方列表是要标注的结构类别。

### 6.3 窗口和默认项目名

建议：

- 窗口标题从 `AntSleap Workbench` 改为 `TaxaMask Workbench`。
- 无项目兜底名从 `Formica_Seg` 改为 `TaxaMask_Project`。
- 内部 Python 包名、模块路径和历史类名不改，例如 `AntSleap` 包继续保留，避免破坏导入、旧项目和开发上下文。

风险：

- 对外 UI 名称和内部工程名会短期并存，需要在开发文档里说明：这是为了兼容历史项目，不代表用户界面仍按蚂蚁专用产品定位。

## 7. 导出调整

### 7.1 COCO

当前：

```json
"supercategory": "ant_part"
```

建议：

```json
"supercategory": "biological_structure"
```

或项目级可配置：

```json
"category_supercategory": "ant_part"
```

优先级建议：

1. 如果项目模板或项目配置提供 `category_supercategory`，使用它。
2. 否则默认 `biological_structure`。

### 7.2 YOLO

YOLO 导出本身基本通用，主要检查：

- `dataset.yaml` 是否正确写入用户自定义 taxonomy。
- 类名是否需要安全转义或引用，避免空格、冒号等引起下游 YAML 解析问题。

当前已有 part name 安全检查，但导出 YAML 仍建议用 JSON/YAML 库或简单引号保护。

### 7.3 Multimodal JSONL

当前：

```json
"schema_version": "formica-multimodal-sample-v1"
```

建议改为：

```json
"schema_version": "taxamask-multimodal-sample-v1"
```

并新增占位字段：

```json
"taxon": "...",
"taxon_rank": "",
"taxon_metadata": {},
"annotation_status": "researcher_verified",
"review_status": "manual_or_auto_pending"
```

注意：不要把 PDF 自动候选直接伪装成已人工确认真值。标注工作台导出的数据可以更接近训练真值，但仍应保留复核状态。

## 8. 路径恢复逻辑

当前 `_resolve_known_relocated_output()` 写死 `Formica-Flow_output` 和 `C:\savedata\Formica-Flow_output`。

建议短期策略：

- 保留函数，改名或注释为 legacy recovery。
- 不在 README 中对普通用户强调。

建议中期策略：

- 增加配置项 `known_relocated_roots`。
- 让用户在本地配置中写旧 root 与新 root 的映射。

这不影响标注工作台通用物种能力，但影响开源代码观感。

## 9. Blink 与级联路由

Blink 工作台的“父部位 -> 子部位”逻辑本身可以跨物种，但文案应更中性。

建议表达：

- 当前技术含义是“上下文区域 -> 目标结构”。
- 对蚂蚁可以理解为 Head -> Mandible。
- 对植物可以理解为 Flower -> Stamen。
- 对其他图像可以理解为 Whole specimen -> Diagnostic region。

建议 UI / 文档逐步把 `parent/child` 解释为 context / target，底层字段可暂时保持不变。

## 9.5 模型通用性判断

当前训练链路应表述为“模型架构可跨物种训练”，而不是“已有权重天然通用于所有物种”。

### Locator

Locator 当前使用 `TraitRegressor`，本质是多通道热图定位 + 宽高回归：

- 输入是 RGB 图像。
- 输出通道数等于当前项目的 `locator_scope` 数量。
- 每个输出通道对应一个 locator scope 标签。
- 训练样本来自当前项目中有标注的图片。

结论：

- 架构是通用的，不写死蚂蚁形态。
- 权重不是通用的。Locator 权重和训练时的 `locator_scope`、标签顺序、图像域强绑定。
- 换成植物、鱼类、甲虫或其他图像域后，应重新训练 Locator，不能把蚂蚁 Locator 权重当作通用定位器直接使用。
- 如果只改 taxonomy 数量，程序会重建 Locator 输出通道，并提示需要重新训练或选择匹配模型。

### SAM / Segmenter

SAM 使用 `sam_b.pt` 作为基础模型，训练时冻结 image encoder 和 prompt encoder，只训练 mask decoder：

- Base SAM 是通用分割基础模型。
- 微调后的 decoder 会吸收当前项目的图像域和 mask 风格。
- 训练时不直接传入类别文本，SAM 主要根据 box prompt 和图像内容生成 mask。

结论：

- Base SAM 更接近通用工具，适合不同物种作为起点。
- Fine-tuned SAM decoder 是项目/数据域适配模型，不应假设蚂蚁微调权重能提升其他物种。
- 对其他类群，推荐先用 Base SAM 做辅助标注；如果目标图像域稳定，再用该类群自己的标注微调 decoder。

开源表述建议：

> TaxaMask provides a taxonomy-configurable training pipeline. The model architecture can be trained for different biological structures, but trained locator and fine-tuned SAM decoder weights are project- and taxonomy-specific.

## 9.6 外部模型训练与推理适配

只支持外部预测结果导入是不够的。TaxaMask 的核心价值之一是形成闭环：

```text
人工标注 / SAM 辅助标注
-> 训练当前项目模型
-> 用模型批量预标注
-> 人工复核修正
-> 再训练强化
-> 导出可信数据集
```

如果外部模型只能导入结果，却不能进入训练闭环，那么对拥有自己模型路线的用户来说，TaxaMask 只能变成复核工具，不能成为持续强化的研究工作台。

因此更合适的设计不是“我们替所有模型写训练代码”，而是提供一个稳定的模型适配协议。

### 9.6.1 内置模型作为默认训练后端

当前内置训练后端继续保留：

```text
Built-in Locator + SAM
```

它负责：

- 根据项目 `locator_scope` 训练定位器。
- 根据项目 `taxonomy` 训练 SAM decoder。
- 训练后权重进入现有模型选择下拉框。
- 批量标注继续走当前内置推理链。

这条链是默认体验，也是蚂蚁模板验证最充分的路线。

### 9.6.2 外部脚本后端协议

新增外部模型后端时，TaxaMask 不直接理解每个模型内部结构，而是要求用户提供符合契约的训练脚本和推理脚本。这个入口主要面向有一定模型基础的用户；他们可以把契约直接提交给大模型，请大模型生成或改写符合自己模型需求的脚本，然后人工确认后替换到 TaxaMask 中。

TaxaMask 侧只识别三个标准动作：

```text
prepare_dataset
train
predict
```

建议后端配置示例：

```json
{
  "backend_id": "custom_yolo_leaf_detector",
  "display_name": "Custom YOLO Leaf Detector",
  "backend_type": "external_script",
  "tasks": ["detect", "segment"],
  "python_executable": "python",
  "prepare_dataset_command": "{python} scripts/prepare_dataset.py --contract {contract_json}",
  "train_command": "{python} scripts/train_model.py --contract {contract_json}",
  "predict_command": "{python} scripts/predict_image.py --contract {contract_json}",
  "prediction_format": "taxamask_prediction_v1",
  "model_manifest": "{run_dir}/model/taxamask_model_manifest.json"
}
```

这样用户可以保留自己的训练脚本，但 TaxaMask 仍能知道：

- 怎么把当前项目标注导出给它训练。
- 怎么启动训练。
- 训练后的模型文件在哪里。
- 怎么调用该模型预测当前图片或批量图片。
- 预测结果如何导回 TaxaMask 项目。

### 9.6.3 LLM 可提交脚本契约

为了降低高级用户适配门槛，TaxaMask 应提供一份可以直接复制给大模型的契约文本。用户只需要告诉大模型“把我的训练代码改成符合这个契约”，大模型就能生成 `prepare_dataset.py`、`train_model.py`、`predict_image.py` 或单文件脚本。

契约文件建议放在：

```text
docs/contracts/external_backend_contract_v1.md
```

契约核心内容：

```text
你正在为 TaxaMask 编写外部模型后端脚本。

脚本必须接受：
  --contract <contract_json>

contract_json 中包含：
  action: prepare_dataset | train | predict
  project_json: 当前 TaxaMask 项目 JSON
  taxonomy: 当前项目结构标签列表
  locator_scope: 当前粗定位标签列表
  run_dir: 本次运行目录
  dataset_dir: 训练数据输出目录
  model_dir: 模型输出目录
  image_path: 当前预测图片，仅 predict 时存在
  prediction_json: 预测结果输出路径，仅 predict 时存在
  model_manifest: 模型 manifest 输出或读取路径
  log_dir: 日志目录

prepare_dataset 必须：
  读取 project_json
  根据项目 labels 导出训练数据到 dataset_dir
  写 dataset_manifest.json

train 必须：
  读取 dataset_dir 和 dataset_manifest.json
  训练模型
  将模型文件写入 model_dir
  写 taxamask_model_manifest.json

predict 必须：
  读取 image_path 和 model_manifest
  输出 taxamask_prediction_v1 JSON 到 prediction_json
```

这样做的意义：

- 用户不需要理解 TaxaMask 内部类名。
- 用户可以用大模型快速适配自己的 YOLO、Detectron、Mask R-CNN、SAM2 或其他训练代码。
- TaxaMask 的维护边界清楚：我们维护契约和运行器，不维护每个外部模型框架。

### 9.6.4 运行目录与契约 JSON

每次外部训练或推理都必须创建独立运行目录，避免污染项目根目录：

```text
external_runs/
```

### 9.6.5 标准预测结果格式

外部模型至少需要输出：

```json
{
  "schema_version": "taxamask_prediction_v1",
  "image_path": "image.jpg",
  "boxes": {
    "Leaf": [120, 80, 500, 430]
  },
  "polygons": {
    "Leaf": [[120, 80], [500, 90], [480, 430]]
  },
  "scores": {
    "Leaf": 0.91
  },
  "model_id": "custom_yolo_leaf_detector/run_20260503"
}
```

规则：

- 有 polygon 时直接作为候选 mask。
- 只有 box 时，可以进入 SAM 或由用户人工复核。
- 标签必须属于当前项目 taxonomy。
- 如果模型返回了项目不存在的标签，导入时进入警告或忽略，不自动扩展 taxonomy。

### 9.6.6 标准模型 manifest

无论内置模型还是外部模型，建议都写一个 manifest：

```json
{
  "schema_version": "taxamask_model_manifest_v1",
  "model_id": "builtin_locator_sam/20260503_1030",
  "backend_id": "builtin_locator_sam",
  "model_role": "locator_segmenter",
  "taxonomy": ["Head", "Mesosoma", "Gaster"],
  "locator_scope": ["Head", "Mesosoma", "Gaster"],
  "label_order": ["Head", "Mesosoma", "Gaster"],
  "input_size": [512, 512],
  "trained_project": "project.json",
  "created_at": "2026-05-03T10:30:00",
  "weights": {
    "locator": "locator_20260503_1030.pth",
    "segmenter": "sam_decoder_lora_20260503_1030.pth"
  }
}
```

用途：

- 防止标签数量相同但语义顺序错位。
- 防止拿植物权重加载到蚂蚁项目，或反过来。
- 让模型下拉框显示更可靠的信息。
- 为后续开源用户共享模型提供最低限度的可审计元数据。

### 9.6.7 UI 设计

AI 工作流区建议从固定按钮升级为后端可选：

```text
Model Backend:
  Built-in Locator + SAM
  External Backend: Custom YOLO Leaf Detector

Train Models
Auto Current
Batch All
Clear AI Labels
```

当选择内置后端：

- `Train Models` 使用当前内置训练流程。
- `Auto Current / Batch All` 使用当前 Locator + SAM。

当选择外部后端：

- `Train Models` 调用该后端的 `prepare_dataset_command` 和 `train_command`。
- `Auto Current / Batch All` 调用该后端的 `predict_command`。
- 预测 JSON 导入项目，进入同一套人工复核和导出流程。

如果外部后端没有提供 `train_command`：

- UI 可以显示“该后端仅支持推理导入”。
- 但这不能作为 TaxaMask 的主要外部模型策略，只能是兼容模式。

### 9.6.8 干净让位与进程隔离

当用户选择外部后端时，TaxaMask 必须干净让位，不能让内置 Locator / SAM 训练或推理残留进入同一次任务。

训练让位规则：

- 外部训练时不创建 `TrainingThread`。
- 不调用 `engine.train_epoch()`。
- 不调用内置 `TwoStageDataset` 训练链，除非外部脚本自己选择复用导出的数据。
- 不保存 `locator_*.pth` 或 `sam_decoder_lora_*.pth`。
- 只创建 external run directory、contract JSON，并启动外部训练脚本进程。
- 训练完成后只读取外部脚本写出的 `taxamask_model_manifest.json`。

推理让位规则：

- 外部推理时不调用 `engine.predict_full_pipeline()`。
- 不调用当前内置 Locator。
- 不调用内置 SAM，除非用户选择的外部后端明确返回“box only and use TaxaMask SAM postprocess”，该模式必须单独显示。
- `Auto Current` 和 `Batch All` 只启动外部 predict script。
- TaxaMask 只读取 `taxamask_prediction_v1` 并导入项目。

进程管理规则：

- 每次训练/推理只允许一个活动 backend job。
- 启动前禁用对应按钮，结束后恢复。
- 捕获 stdout / stderr 到 `logs/stdout.log` 和 `logs/stderr.log`。
- UI 显示最后若干行日志，完整日志可打开。
- 用户点击停止时，先请求外部进程正常退出；超时后再终止进程树。
- 进程结束后必须清理进程句柄，避免下一次任务误判仍在运行。
- 外部脚本报错时，TaxaMask 不尝试自动修复，只展示日志和 contract 路径，让用户或大模型据此修改脚本。

状态隔离规则：

- 内置模型选择下拉框和外部模型 manifest 下拉框分开。
- 外部后端训练出的模型不混入 `weights/locator_*.pth` / `weights/sam_decoder_lora_*.pth` 列表。
- 外部预测导入的标签仍标记为 `Auto-Annotated` 或更明确的 `External-Auto-Annotated`，供人工复核。
- 任何外部后端结果都不得自动覆盖人工标注，除非用户显式选择覆盖模式。

### 9.6.9 设计边界

TaxaMask 需要负责：

- 定义训练数据导出契约。
- 定义预测结果导入契约。
- 生成可提交给大模型的脚本适配契约。
- 启动外部训练/推理脚本。
- 记录运行日志、模型 manifest 和错误信息。
- 把预测结果放回项目，支持人工复核和再训练。
- 在外部后端运行时干净让位，不混入内置训练或推理线程。

TaxaMask 不需要负责：

- 内置支持所有外部模型框架。
- 替用户维护每个模型的 conda 环境。
- 理解每个外部模型的内部权重结构。
- 保证任意外部模型输出具有生物学可信度。

这样闭环不会断，同时维护边界也清楚。

## 11. 推荐实施顺序

建议分两轮完成。

第一轮，开源门面与导出污染修复：

1. C：COCO / Multimodal schema 清理。
2. B：Taxon 字段兼容和 GUI 文案。
3. D：窗口标题、默认项目名。

第二轮，真正模板化：

1. A：项目模板。
2. F：通用项目测试。
3. E：legacy 路径恢复配置化。

第三轮，外部模型闭环：

1. G：外部脚本后端适配层。
2. 增加 dummy backend 测试和示例契约。
3. 文档中给出“把契约交给大模型生成脚本”的适配流程。

原因：

- 第一轮能最快解决外部用户最容易看到、最容易质疑的问题。
- 第二轮涉及新建项目流程和模板选择，UI 改动稍大，适合验收后再做。
- 第三轮涉及进程管理和脚本契约，必须单独做，避免外部训练/推理和内置训练/推理状态混在一起。

## 12. 不建议本轮做的事

- 不建议重命名 Python 包 `AntSleap`。这会影响导入、路径、已有项目和文档，收益不如窗口标题和导出 schema 清理明显。
- 不建议删除蚂蚁默认模板。它是项目最有研究验证的核心样例。
- 不建议一次性引入完整 Darwin Core schema。长期值得做，但本轮容易把标注工作台改得过重。
- 不建议把所有 `part` 命名立即改成 `structure`。底层字段可先保留，界面和导出语义先通用化。

## 13. 风险

1. 旧项目兼容风险  
   如果直接把 `genus` 删除，旧项目和描述数据库查询会断。必须先做兼容层。

2. 模板选择增加 UI 复杂度  
   对研究者来说，新建项目流程不能太复杂。模板选择应尽量是一个简单下拉框。

3. 旧权重与 taxonomy 绑定  
   换 taxonomy 后旧 Locator 权重不一定能用。界面需要继续提示“Locator scope changed, retrain or select matching model”。

4. 过度通用化风险  
   如果把蚂蚁示例藏得太深，反而削弱当前最可信的应用场景。更好的做法是“蚂蚁模板清晰、通用模板也清晰”。

## 14. 验收口径

用户验收时建议重点看：

1. 这个规划是否符合项目定位：先稳住蚂蚁研究，同时给其他类群清楚入口。
2. 是否接受 `Taxon` 作为 `genus` 的上层通用字段。
3. 是否接受窗口标题和默认项目名改为 TaxaMask。
4. 是否接受 COCO `supercategory` 从 `ant_part` 改为 `biological_structure`。
5. 是否接受 Multimodal JSONL schema 改为 `taxamask-multimodal-sample-v1`。

验收通过后，可按阶段 A-F 进入代码实施。
