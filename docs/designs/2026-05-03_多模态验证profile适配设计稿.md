# 多模态验证 Profile 适配设计稿

日期：2026-05-03

状态：设计方向已确认，待按实施清单进入代码调整

修订说明：2026-05-03 用户进一步指出，多模态验证不是默认开启，必须配置模型后才会运行；同时图文提取阶段本身也需要确认是否具备多物种适配。本设计据此扩展为“图文提取与多模态复核 profile 化”，而不是只改多模态模型提示词。

## 1. 设计目标

当前 PDF 文献筛选已经进入 V2 profile 模式，用户可以把“蚂蚁新种筛选”复制并改成植物、甲虫或其他分类群筛选逻辑。

但多模态验证模块目前还没有同等能力。它能配置 API、模型、批量大小和阈值，却不能配置“我现在要复核什么类型的分类学图版”。核心判断仍然围绕“单一物种蚂蚁三视图整图”。

本设计的目标是：把 PDF 后半段的 figure 提取、文本证据组装和多模态 figure 复核一起改成 profile 化，使用户可以保留蚂蚁三视图作为示例，同时复制出其他生物类群或其他图版类型的提取与复核逻辑。

## 2. 当前程序在做什么

现有图文提取与多模态验证链路：

1. GUI 或 CLI 启动 PDF figure 提取。
2. 提取器生成 figure 候选图，以及 caption、附近正文、物种核心文本、扩展文本。
3. 提取器根据 species 名称、分类学关键词、章节标题和页面距离组装分层文本证据。
4. 如果启用了真实多模态验证，`MultimodalValidator` 把候选图和文本证据送给多模态模型。
5. 如果没有启用或模型不可用，程序使用 mock/default 复核路径，把候选放入人工复核队列。
6. 模型或 mock 返回每张候选图是否接受、置信度、疑似物种、检测到的视图和拒绝原因。
7. 程序按 `accept_threshold`、比较图标记、多物种标记和必须视图决定进入 accepted、review 或 rejected。

现有限制：

- 图文提取阶段的分类学关键词包含 `formicidae`、`ant`、`worker`、`queen`、`male` 等蚂蚁/昆虫语境词。
- 文本证据选择依赖固定的 `CORE_SECTION_HINTS` 和 `EXTENDED_SECTION_HINTS`，尚不能由用户按类群调整。
- 最终接受条件里固定要求 `lateral`、`dorsal`、`head_frontal` 三视图。
- system prompt 写死为蚂蚁分类学图像审稿助手。
- 接受标准写死为单一物种蚂蚁三视图整图。
- 视图字段写死为 `lateral`、`dorsal`、`head_frontal`。
- mock 回退逻辑也依赖 Formicidae、ant、worker 和三视图关键词。
- GUI 没有让用户选择“图文提取与多模态复核 profile”的入口。

## 3. 为什么这影响多物种适配

对真实研究流程的影响是：

- 筛植物图版时，程序仍会问模型“是不是蚂蚁三视图”，容易误拒绝叶、花、果实、种子、显微结构图。
- 筛甲虫或其他昆虫时，三视图字段可能不够用，例如背面、腹面、侧面、生殖器、翅脉、触角等结构需要不同命名。
- 筛真菌、鱼类、植物或化石图版时，“worker / queen / male / full-face”等蚂蚁词会变成噪音。
- 如果真实多模态调用失败，mock 回退仍然只会按蚂蚁三视图估计，不能作为跨物种可信判断。

因此当前状态应该表述为：图文提取有一部分版面算法是通用的，但证据组装、接受条件和多模态复核逻辑仍明显偏蚂蚁三视图；多模态验证已支持可配置运行环境，但尚未支持可配置分类学复核逻辑。

## 4. 目标用户体验

用户反馈：当前界面的“逻辑方案”区域位于 PDF 筛选和多模态验证 / 数据提取区域上方。因此多模态验证 profile 最好也放在这个顶部方案区，而不是藏在 figure 提取区域深处。

推荐界面关系：

```text
Logic Profiles

PDF Screening Profile:
[蚂蚁新种筛选_V2示例 ▼] [Advanced Screening Settings]

Figure Extraction / Review Profile:
[蚂蚁三视图提取复核_示例 ▼] [Advanced Figure Settings]
```

这里的关键点是：两个栏位并列，但不要合并成同一个 JSON。

- PDF Screening Profile 管“哪些 PDF 文献值得进入后续提取”。
- Figure Extraction / Review Profile 管“从 PDF 中怎样提取 figure 候选、怎样组装图文证据、以及启用多模态时怎样判断候选是不是目标分类学图版”。

这样设计对研究流程更清楚：上游先筛文献，下游再提取和复核图版。用户可以让两个 profile 指向同一个研究对象，例如都选择蚂蚁；也可以在测试时分别调整，例如文献筛选先宽一点，图文提取与图版复核再严格一点。

这个下游 profile 即使在未开启真实多模态验证时也应该生效，因为图文提取阶段本身就需要知道当前目标类群、核心章节、分类学关键词和期望图版结构。真实多模态验证只是在这个 profile 基础上的进一步复核。

不建议再额外新增第三个并列栏位，例如单独的 `Multimodal Validation Profile`。原因是多模态验证复核的是同一批 figure 候选和同一套文本证据，它应该继承图文提取阶段的目标类群、期望结构、排除规则和接受标准。更合适的结构是：

```text
Figure Extraction / Review Profile
  - Extraction Rules：控制候选图、caption、附近文本、核心证据和扩展证据怎样提取。
  - Review Rules：控制启用多模态模型时，怎样判断候选是否属于目标分类学图版。
```

界面上的 `Advanced Figure Settings` 可以用两个页签或分组呈现：

```text
[Extraction Rules] [Multimodal Review Rules]
```

这样用户只需要选择一个下游 profile。不开多模态验证时，只使用 Extraction Rules；开启多模态验证且模型配置可用时，再额外使用 Multimodal Review Rules。

需要独立保留的是 API / 模型运行设置，而不是再做一个生物学逻辑 profile。也就是说：

- Figure Extraction / Review Profile：管“生物学目标和图文逻辑”。
- API / Model Settings：管“用哪个模型、base_url、api_key、batch、timeout”。

这两个东西性质不同，应该分开。前者适合随项目或开源模板保存；后者通常是本机私密运行配置，不应该和开源 profile 混在一起。

API / Model Settings 建议再按任务角色分成两类，而不是只有一组全局输入：

```text
Text LLM API
  用途：PDF 文献筛选、标题/摘要/正文文本判断、批量 include/exclude/uncertain。
  常见模型：文本模型或便宜高速的通用对话模型。

Multimodal LLM API
  用途：figure 图像 + caption + species_core/species_extended 复核。
  常见模型：支持图像输入的视觉语言模型。
```

界面上可以做成：

```text
API / Model Settings

[Text LLM]
Base URL: [...]
Model: [...]
API Key: [...]
Protocol: [...]

[Multimodal LLM]
[ ] Use same provider as Text LLM
Base URL: [...]
Model: [...]
API Key: [...]
Protocol: [...]
Image Detail: [...]
```

默认体验建议：

- 初次使用时，`Multimodal LLM` 可以勾选 `Use same provider as Text LLM`，减少配置负担。
- 一旦用户取消勾选，就可以给多模态验证单独填 base_url、model 和 key。
- 如果只做 PDF 文本筛选，只需要配置 Text LLM。
- 如果要开启真实多模态验证，必须配置 Multimodal LLM，且模型需要支持图像输入。
- API key 不写入开源 profile；如需保存，只进入本机 runtime settings，并默认不记住 key。

这样设计的实际意义是：用户可以用便宜、快的文本模型筛文献，再用更强但更贵的视觉模型复核 figure。两种任务的成本和能力需求不同，分开后更容易控制批量处理成本，也更容易排查“文本筛选能跑，但多模态看图失败”的问题。

实际使用方式：

1. 默认提供“蚂蚁三视图提取复核_示例”。
2. 用户复制该 profile。
3. 把目标类群、接受标准、视图字段、排除规则和提示词改成自己的对象。
4. 小批量运行 5-20 篇 PDF。
5. 检查 accepted / review / rejected 的分布和模型理由。
6. 稳定后用于大批量处理。

## 5. 建议新增目录

新增：

```text
multimodal_configs/
```

用途：存放 figure 提取与多模态复核 profile。

初始示例：

```text
multimodal_configs/蚂蚁三视图提取复核_示例.json
multimodal_configs/通用分类学图版提取复核_模板.json
multimodal_configs/植物分类学图版提取复核_模板.json
```

`screener_configs/` 继续只负责 PDF 文献筛选。`multimodal_configs/` 负责 figure 提取、证据组装和多模态 figure 复核。两者可以在界面上并排展示，但文件目录和数据结构不要混放，否则用户会分不清“筛文献”和“筛图版”。

## 6. Profile 字段草案

建议一个多模态 profile 使用如下结构：

```json
{
  "schema_version": "taxamask-multimodal-review-profile-v1",
  "profile_name": "蚂蚁三视图提取复核_示例",
  "profile_description": "从 PDF 中提取并复核单一物种蚂蚁分类学三视图 figure。",
  "target_taxon": {
    "display_name": "蚂蚁",
    "scientific_scope": "Formicidae",
    "tested_status": "ant-tested"
  },
  "review_goal": {
    "acceptance_goal": "只接受主体为单一物种蚂蚁分类学三视图的整张 figure。",
    "accept_if": [
      "主体是同一物种或同一分类单元",
      "包含分类学图版所需的主要视图",
      "caption 或附近文本能支持物种候选"
    ],
    "reject_if": [
      "多物种比较图",
      "仅分布图、系统树、生态照片或实验图",
      "主体不是目标分类群",
      "图像证据与文字证据明显冲突"
    ]
  },
  "extraction_rules": {
    "caption_patterns": [
      "^Figure\\s+\\d+[A-Za-z]?[:\\.]\\s+.+",
      "^Fig\\.\\s*\\d+[A-Za-z]?[:\\.]\\s+.+",
      "^Plate\\s+\\d+[A-Za-z]?[:\\.]\\s+.+",
      "^图\\s*\\d+[A-Za-z]?[:：\\.]\\s*.+"
    ],
    "core_section_hints": [
      "diagnosis",
      "description",
      "measurements",
      "material_examined",
      "type_material",
      "distribution",
      "remarks",
      "biology",
      "etymology",
      "species_account"
    ],
    "extended_section_hints": [
      "identification_key",
      "key_to_species",
      "taxonomic_treatment",
      "species_account",
      "synoptic_list"
    ],
    "evidence_text_limits": {
      "figure_local_chars": 1200,
      "species_core_chars": 2200,
      "species_extended_chars": 2200
    }
  },
  "view_schema": {
    "required_or_expected_views": ["lateral", "dorsal", "head_frontal"],
    "view_terms": {
      "lateral": ["lateral", "profile", "side view", "body in profile"],
      "dorsal": ["dorsal", "in dorsal view", "from above"],
      "head_frontal": ["full-face", "frontal", "head in full-face"]
    }
  },
  "taxonomy_terms": [
    "formicidae",
    "ant",
    "worker",
    "holotype",
    "paratype",
    "diagnosis",
    "description",
    "material examined",
    "sp. nov"
  ],
  "rejection_terms": [
    "comparison",
    "distribution map only",
    "phylogeny",
    "ecology",
    "behavior"
  ],
  "category_values": [
    "target_taxon_figure",
    "comparison_or_multi_species",
    "non_target_or_other",
    "uncertain"
  ],
  "output_contract": {
    "species_field_label": "species_candidate",
    "detected_parts_field_label": "detected_views"
  },
  "prompt": {
    "system_prompt": "你是严谨的蚂蚁分类学图像审稿助手...",
    "user_instructions": "请按顺序审查以下 figure 候选..."
  },
  "decision_thresholds": {
    "accept_threshold": 0.85
  },
  "mock_fallback": {
    "enabled": true,
    "accept_requires_text_hits": 2,
    "accept_requires_parts": ["lateral", "dorsal", "head_frontal"]
  }
}
```

字段设计原则：

- `target_taxon` 说明目标类群，但不把程序锁死到某一个类群。
- `review_goal` 给用户写自然语言规则，这是模型判断的核心。
- `extraction_rules` 控制图文提取阶段怎样识别 caption、核心章节、扩展证据和文本长度；即使不开多模态验证也应该生效。
- `view_schema` 不一定只能是视图，也可以扩展成结构部位，例如植物的 leaf、flower、fruit、seed。
- `taxonomy_terms` 和 `rejection_terms` 用于 mock 回退、本地提示和审计，不应替代真实多模态模型判断。
- `prompt` 允许高级用户完全改写提示词。
- `decision_thresholds.accept_threshold` 允许不同 profile 使用不同接受阈值。

## 7. 蚂蚁示例应该如何保留

蚂蚁三视图不应该从项目中消失，因为它是当前最真实、验证最多的研究场景。

建议保留方式：

- 默认 profile：`蚂蚁三视图提取复核_示例.json`
- 文档中明确说明：这是可复制的参考模板，不是唯一支持对象。
- 程序内默认值保持和当前行为接近，避免已有用户工作流突然改变。
- README 或用户手册中避免宣称所有类群都已同等验证；应写“蚂蚁场景验证最充分，其他类群可通过 profile 适配并需小批量校验”。

## 8. 通用模板应该怎么写

通用模板不应假装知道所有生物类群的标准图版形式，而应提供一个中性框架：

- 目标：接受“目标分类群的单一分类单元图版”。
- 结构字段：使用 `primary_view_or_structure`、`diagnostic_structure`、`habitus_or_overview` 这类中性命名。
- 拒绝：多分类单元比较图、非目标类群、地图/树/表格/实验曲线、证据不足。
- 结果：不强迫三视图，只要求模型列出检测到的关键结构。

这样其他研究者复制后，可以改成：

- 植物：habit、leaf、flower、fruit、seed、micromorphology。
- 甲虫：dorsal habitus、lateral habitus、ventral view、genitalia、antenna。
- 鱼类：whole body lateral、head detail、fin detail、scale count evidence。
- 真菌：basidiocarp、spore、pileipellis、hymenophore、microscopy plate。

## 9. 代码实施范围

建议分四步实施。

### 阶段 A：配置读取和默认兼容

目标：先让 `EnhancedPDFExtractionSystem` 和 `MultimodalValidator` 能读 profile，但默认行为不变。

修改点：

- 新增 `multimodal_configs/` 示例文件。
- `EnhancedPDFExtractionSystem.__init__` 读取 figure extraction / review profile。
- `MultimodalValidator.__init__` 接收同一 profile 中的 review 部分。
- 将当前写死常量变成 profile 默认值：
  - `CORE_VIEW_TERMS`
  - `TAXONOMIC_TERMS`
  - `REJECTION_TERMS`
  - `AUXILIARY_TERMS`
  - `CORE_SECTION_TERMS`
- `MAX_LOCAL_CHARS`
- `MAX_CORE_CHARS`
- `MAX_EXTENDED_CHARS`
- 未传 profile 时自动使用内置蚂蚁三视图默认值。

验收标准：

- 不传 profile，旧蚂蚁三视图流程结果基本不变。
- 传入 profile 后，日志能显示当前 profile 名称。
- JSON 配置解析失败时给出明确错误，不静默退回错误逻辑。

### 阶段 B：图文提取证据组装 profile 化

目标：即使不启用真实多模态模型，figure 候选和文本证据也能按目标类群调整。

修改点：

- `_score_taxonomic_text()` 使用 profile 的 `taxonomy_terms`。
- `_detect_section_hint()` 使用 profile 的 `extraction_rules.core_section_hints` 和 `extended_section_hints`。
- `_is_valid_figure_title()` 使用 profile 的 `caption_patterns`。
- `_select_local_blocks()`、`_select_species_core_blocks()`、`_select_species_extended_blocks()` 使用 profile 的关键词、章节和文本长度配置。
- `_has_required_triptych_views()` 改成基于 profile 的 `required_or_expected_views`，不再固定三视图。
- `review_status` 的接受条件由 profile 控制，至少支持“必须具备全部结构”和“模型 accept 即可，结构字段只作记录”两种模式。

验收标准：

- 不开多模态验证时，profile 仍影响候选证据组装。
- 蚂蚁 profile 仍能按原三视图逻辑进入 accepted / review / rejected。
- 植物 profile 不会因为缺少 `lateral / dorsal / head_frontal` 被自动拒绝。

### 阶段 C：提示词和 schema profile 化

目标：真实多模态模型调用不再写死蚂蚁三视图。

修改点：

- `_build_review_system_prompt()` 使用 profile 的 `system_prompt`。
- `_build_chat_messages()` 使用 profile 的 `user_instructions`、接受标准、类别值和视图字段。
- `_build_triptych_json_schema()` 改名或内部泛化，不再限定 enum 为三视图。
- `_parse_triptych_results()` 使用 profile 的允许字段规范化 `detected_views` 或 `detected_parts`。

验收标准：

- 蚂蚁 profile 仍返回 lateral / dorsal / head_frontal。
- 通用 profile 能返回用户定义的结构字段。
- 模型必须仍然为每个 candidate_id 返回一条结果。

### 阶段 D：mock 回退逻辑 profile 化

目标：真实多模态调用失败时，回退结果也不要冒充跨物种智能判断。

修改点：

- `_mock_review_candidate()` 使用 profile 的 `view_terms`、`taxonomy_terms`、`rejection_terms`。
- mock 结果默认只进入 Review，不进入可信 accepted。
- mock reasoning 明确写出“真实多模态调用失败后降级”。

验收标准：

- 关闭 API 或 API 失败时，候选进入 Review。
- mock 不因蚂蚁关键词误接受非蚂蚁图版。
- 输出中保留 `multimodal_review_mode = mock`。

### 阶段 E：GUI 和 CLI 接口

目标：用户可以像 PDF 筛选 profile 一样管理 figure 提取 / 多模态复核 profile。

修改点：

- GUI 顶部逻辑方案区新增 `Figure Extraction / Review Profile` 下拉框，与现有 PDF Screening Profile 并列。
- 新增 Advanced Figure Settings 或先提供“打开 profile 文件夹/复制模板”的轻量入口。
- CLI `extract_figures.py` 支持 `--figure-profile`，可以兼容别名 `--multimodal-profile`。
- `--multimodal-config` 继续负责 API 运行参数，`--figure-profile` 负责图文提取和分类学复核逻辑。

验收标准：

- GUI 顶部能同时看到 PDF Screening Profile 和 Figure Extraction / Review Profile。
- GUI 能为图文提取 / 多模态复核选择蚂蚁示例和通用模板。
- 启动 PDF 文献筛选时只使用 PDF Screening Profile。
- 启动 figure 提取时使用 Figure Extraction / Review Profile。
- 启用多模态验证时，同一个 Figure Extraction / Review Profile 的 review 部分继续生效。
- CLI 能单独传 API config 和 review profile。
- 运行日志记录 profile 名称、模型名、阈值和是否真实复核。

## 10. 建议不要做的事

- 不要把 PDF 文献筛选 profile 和多模态图版复核 profile 混在同一个目录。
- 不要只改多模态提示词，而忽略 figure 提取阶段的证据组装逻辑。
- 不要在代码中继续追加隐藏的蚂蚁专用规则。
- 不要让 mock 回退结果直接进入高可信训练集。
- 不要声称“支持所有物种自动准确复核”，应表述为“可通过 profile 适配，并建议小批量验证”。
- 不要一次性重写整个 PDF 提取器，优先把复核逻辑抽出来，减少对现有提取稳定性的影响。

## 12. 最小验收样例

建议实施后至少用三类样例验收：

1. 蚂蚁新种 PDF：应能接受真实单物种三视图，拒绝地图、系统树、多物种比较图。
2. 植物分类 PDF：用植物 profile 时，不应因为缺少 lateral / dorsal / full-face 而拒绝分类图版。
3. 非目标类群 PDF：用蚂蚁 profile 时，植物或甲虫图版不应被误接受为蚂蚁三视图。

## 13. 给开源用户的表述建议

建议在后续 README / 手册中这样表达：

> 多模态 figure 复核支持 profile 化配置。项目默认提供蚂蚁三视图复核示例，因为这是当前验证最充分的场景。其他分类群可复制通用模板并调整目标类群、图版结构、接受标准和提示词；正式大批量使用前建议先进行小批量人工验收。

这既保留了项目从蚂蚁分类学出发的可信根基，也避免对其他类群做过度承诺。
