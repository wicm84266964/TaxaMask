## 0.1 当前实现状态（2026-05-26）

这一轮已经落地为可试用的“草稿优先”功能，但仍建议先做小样本复核，不建议无人值守大规模运行。

已经实现：

- VLM 目标部位由用户从当前项目已有部位中勾选，且与主定位部位 `locator_scope` 分开；
- 处理范围支持 `Current image only` 和 `All imported images`，全项目批量运行前有二次确认；
- 批量处理按图像线性执行，并按每张图约 6 个步骤推进 GUI 进度条；
- 复用 PDF Evidence 的多模态 API 设置；缺少目标部位或 API 配置时，弹窗按钮会直接跳到相应设置界面；

本轮同时修复的相关交互漏洞：

- Blink 父级解析不再使用“已有父级框”或“唯一主部位”作为自动兜底；
- 新建部位默认留在未分组结构中，右侧父级下拉框只提供候选；
- 用户手动选择父级上下文后，才写入 `blink_context_roi_parents` 并注册 `parent -> child` candidate route；

- 曾出现 `Expecting value: line 1 column 1 (char 0)`，原因是多模态返回空响应或非 JSON；

- 试跑时曾选到 `Object / Region / Structure` 这类通用部位，建议真实蚂蚁项目中优先勾选 `Head / Mesosoma / Gaster / Petiole / Eye / Mandible` 等具体结构，避免 prompt 目标过泛。

## 1. 背景

现在 2D/STL 主标注工作台和 Blink 父子精修已经合并到同一条日常工作线中。

现有流程对已经有少量标注素材后的迭代比较友好：

```text
人工标注一批
-> 训练 Locator / SAM / Blink route expert
-> 自动预标注更多图片
-> 人工复核和修正
-> 再训练
```

真正痛苦的是最开始没有任何 Locator/Blink 素材的阶段，也就是“第一公里”：

```text
没有训练样本
-> Locator 还不能可靠定位
-> Blink route expert 还没有训练数据
-> SAM 需要用户手工给 box prompt
-> 用户必须全程手工框选常见部位
```

用户已经用世界前沿多模态大模型做过一次实践。观察结果是：在整张蚂蚁分类学图片上叠加细网格后，大模型可以根据视觉位置为常见结构给出相对可用的提示框，例如：

- Head / 头部
- Mesosoma / 胸部
- Gaster / 腹部
- Petiole / 腹柄节
- Eye / 眼睛
- Mandible / 上颚/大颚

这些框不应被理解为最终分类学标注，但很适合替代“给 SAM 画第一批提示框”的重复劳动。

## 2. 设计目标

### 2.1 程序要做什么

新增一个 VLM 第一公里预处理能力：

```text
当前图片或一批图片
-> 生成带网格的临时图
-> 调用多模态大模型识别常见部位位置
-> 得到候选 box
-> 用候选 box 驱动 SAM 生成草稿 polygon
-> 写回主标注工作台，供人工复核
```

### 2.2 为什么对研究流程重要

它解决的是没有任何定位器训练材料时的启动成本。

对研究者来说，实际收益是：

- 不再需要从第一张图开始完全手工给每个常见部位画 SAM 提示框；
- 可以先让 VLM 给出常见结构的候选框；
- SAM 根据这些框生成初稿轮廓；
- 研究者把时间集中在复核、修正和确认上；
- 第一批确认后的标注再用于训练真正属于本项目的 Locator/SAM/Blink。

### 2.3 关键边界

VLM 不是训练真值来源。

VLM 在这里只做：

```text
候选框建议器
```

不做：

```text
最终分类学判断器
训练数据自动确认器
取代人工复核的标注者
```

## 3. 非目标

第一轮不做以下事情：

- 不让 VLM 输出直接进入训练集；
- 不让 VLM 自动覆盖人工标注；
- 不替代 Blink route expert；
- 不把 VLM 做成新的主模型后端；
- 不要求用户配置新的 API 密钥体系，优先复用 PDF Evidence 的 Multimodal LLM API；
- 不默认批量跑完整项目，先从当前图像和小批量开始；
- 不把支持范围宣传为通用所有动物或所有分类群，先稳定服务蚂蚁分类学图片。

## 4. 核心设计原则

### 4.1 三层标注状态

为了保护训练数据可信度，必须把三层状态分开。

第一层：VLM 候选框

```text
来源：多模态大模型
用途：给 SAM 的提示框 / 画布上的橙色草稿框
训练资格：无
```

第二层：SAM 草稿 polygon

```text
来源：SAM 根据 VLM box 生成
用途：给研究者复核的初稿轮廓
训练资格：默认无，仍标记为 Auto-Annotated
```

第三层：人工确认标注

```text
来源：研究者复核、修正、确认
用途：训练 Locator/SAM/Blink
训练资格：有
```

### 4.2 不污染训练链

程序行为：

- VLM 结果默认写成 AI 草稿；
- SAM 生成的 polygon 默认标记为 `Auto-Annotated`；
- 训练预检默认跳过仍未复核的 `Auto-Annotated` 草稿；
- 用户完成复核动作后，才移除草稿标记，使其成为可训练材料。
- 复核动作可以是单个部位空格确认，也可以是当前图像的一键通过。
- VLM 候选框本身不单独作为训练材料；但它可以作为已确认 SAM polygon 的来源框保留下来，方便追溯和显示。
- 即使框的位置看起来准确，也必须经过人工复核动作后，才能让对应的 SAM polygon 成为可训练标注。

为什么重要：

- 大模型有时会把相邻结构混淆；
- 分类学照片可能有遮挡、局部缺失、比例尺、插图或多视图；
- 如果草稿自动进入训练，错误会被模型学习并在后续批量推理中放大。

取舍：

- 需要用户至少做一次审核动作；
- 但可以避免“看起来自动化很快，实际训练集已经混入未审错误”的风险。

### 4.3 复核优先级：人工结果覆盖 AI 草稿

程序行为：

- VLM 候选框显示为橙色 `aibox`；
- SAM 根据 `aibox` 生成的 polygon 仍属于 AI 草稿；
- 用户检查后，如果草稿正确，可以沿用当前空格确认机制，把当前部位从草稿转为已确认标注；
- 如果当前图像上一批草稿整体没有问题，用户可以使用“一键通过当前图像 AI 草稿”把剩余草稿整体确认为可训练标注；
- 一键通过只对当前图像生效，不提供跨图像批量通过；
- 用户如果认为框或 polygon 不准确，直接重新框选当前部位，并用新框调用 SAM；
- 人工重新框选完成后，程序自动移除该部位对应的橙色 `aibox` 和旧的 AI SAM polygon；
- 同一部位同时存在人工结果和 AI 草稿时，以人工结果为准。

为什么重要：

- 研究者不需要进入复杂的“接受/拒绝表格”；
- 没有明显问题的草稿不需要逐个点击，避免确认成本抵消自动化收益；
- 错误草稿的修正方式和现有 SAM 框选习惯一致；
- 训练集只吸收研究者明确确认或重新标注过的结果。

取舍：

- 程序需要维护 `aibox` 与其生成的 SAM polygon 之间的来源关系；
- 但换来的是更自然的操作：少量错误就重画，整体可用就一键通过。

### 4.4 VLM 只负责框，不负责最终轮廓

程序行为：

- VLM 输出 box；
- SAM 输出 polygon；
- 人工复核 polygon。

为什么重要：

- 多模态大模型擅长在图像中说出大致位置；
- SAM 擅长根据 box 生成边界；
- 分类学研究者擅长判断这个边界是否符合真实结构。

取舍：

- VLM 不能直接解决所有标注；
- 但它能替代最重复的“第一批框选”劳动。

## 5. 推荐工作流

### 5.1 当前图像试跑

```text
打开 2D/STL 主标注工作台
-> 选择一张蚂蚁分类学图片
-> 点击右侧栏上方的 VLM 预标注入口
-> 程序生成网格图并调用多模态模型
-> 候选框显示为橙色 aibox
-> SAM 尝试生成草稿 polygon
-> 用户逐个检查
-> 正确则按空格确认
-> 不正确则重新框选当前部位并调用 SAM
-> 程序移除被替换的 aibox 和旧 AI polygon
```

适用场景：

- 第一次验证某个模型/provider 是否适合这类图片；
- 检查 VLM 在当前图版风格上的稳定性；
- 避免一开始就批量制造大量低质量草稿。

### 5.2 小批量预处理

```text
选择 5-20 张未标注图片
-> 批量生成 VLM 候选框
-> 保存每张图的网格图、原始响应、解析报告
-> 写入项目草稿层
-> 用户按图复核
```

适用场景：

- 快速建立第一批训练样本；
- 让人工标注工作从“从零画框”变成“检查和修正草稿”。

### 5.3 进入传统训练循环

```text
复核确认第一批草稿
-> 训练 Locator/SAM
-> 使用内置自动标注或 Blink route expert
-> 继续人工复核和训练迭代
```

VLM 主要服务最早期阶段。等项目自己的定位器和专家模型可用后，日常工作应回到本地模型辅助标注。

## 6. GUI 入口设计

### 6.1 推荐入口位置

放在主标注工作台右侧栏上方，与“询问 Agent”同一列按钮保持一致。

按钮建议命名：

```text
VLM 预标注
```

英文可为：

```text
VLM Pre-Annotate
```

按钮行为：

```text
按当前设置运行 VLM 第一公里预标注。
如果设置为当前图像，则只处理当前图像。
如果设置为已导入所有图像，则执行批量预处理。
运行结果写入草稿层，不进入训练。
```

为什么放这里：

- 右侧栏上方是研究者日常操作区，入口足够明显；
- 与 Agent 入口同列，可以表达“这是 AI 辅助动作”，但不混入 Blink 的父子精修逻辑；
- 点击入口后直接按设置执行，避免每次都弹出复杂参数窗口。

### 6.2 设置页中的 VLM 预标注选项

建议在现有设置中增加一个 `AI 多模态预标注` 区域，而不是另做一个大型独立界面。

这个区域包含：

```text
启用部位选择：
从当前项目已经创建的部位中勾选哪些部位交给 VLM 预标注

处理范围：
- 当前图像
- 已导入的所有图像

草稿写回：
- 保留 VLM 候选框为橙色 aibox
- SAM 成功时生成 Auto-Annotated polygon

安全提示：
这些结果必须经过人工审核动作，未确认前不会进入训练
```

与现有“主部位”设置的关系：

- 现有主部位勾选机制可以作为交互参考，但 VLM 预标注部位必须单独维护；
- VLM 预标注部位应使用当前项目已有部位，而不是默认所有部位；
- 不应强制等同于主部位，因为多模态模型有时也能预标注非主部位，例如 Eye、Mandible、Scape 等；
- 程序可以提供常见蚂蚁结构作为建议列表；
- 实际送给 VLM 的目标部位必须由用户在设置中自行勾选；
- 用户可以手动增减，例如只让 VLM 尝试 `Head / Mesosoma / Gaster / Eye / Mandible`。

为什么这样设计：

- 多模态模型对常见清晰结构更稳定，对非常细碎或项目特有结构不一定可靠；
- 让用户选择部位，可以避免把模型送去猜不适合它判断的结构；
- 当前图像和全项目批量的选择放在设置中，能避免误触后批量调用 API。

取舍：

- 设置页会多一个区域；
- 但日常入口保持简单，适合频繁使用。

### 6.3 为什么不放在 Blink 面板内部

程序层面：

- VLM 第一公里主要解决“父级/常见结构还没有框”的问题；
- Blink 面板主要处理已经有父级上下文后的父子精修；
- 二者相邻但职责不同。

研究流程层面：

- 先用 VLM 建立 Head/Mesosoma/Gaster/Eye/Mandible 等初始框；
- 再用 Blink 做父子路由、子部位精修和专家训练。

取舍：

- 放在右侧栏上方更清楚：它属于 AI 辅助预处理入口；
- Blink 面板保持专注：它属于父子精修。

### 6.4 用户可见提示

按钮 tooltip 或短提示应说明：

```text
使用已配置的多模态模型为当前图片生成 SAM 提示框。
结果在复核前只是 AI 草稿，不会直接作为训练真值。
```

## 7. API 设置复用

### 7.1 程序行为

复用 PDF Evidence 工具中已有的 `Multimodal LLM API` 设置：

- API key；
- base_url；
- model；
- api_protocol；
- image_detail。

如果用户勾选 “Use same provider as Text LLM”，则复用文本 LLM provider，但模型必须支持图像输入。

### 7.2 为什么重要

研究者不应该为了同一个多模态模型在不同页面重复配置密钥。

### 7.3 取舍

- 好处：降低配置成本；
- 代价：需要在 UI 上提示用户先配置多模态 API；
- 风险：某些 text LLM provider 不支持图像输入，必须在报错时说明是 provider/model 能力问题。

## 8. 网格图方案

### 8.1 程序行为

对每张输入图生成一个临时网格图：

```text
原图
-> 按最长边缩放到适合 VLM 输入的尺寸
-> 叠加细网格线
-> 在上边和左边标注坐标编号
-> 发送给 VLM
```

默认网格建议：

```text
12 x 12
```

### 8.2 为什么选择网格

多模态模型对“用自然语言描述位置”容易含糊，例如“头部左侧偏上”。网格坐标可以把大模型的视觉判断转换为可解析的 box。

### 8.3 坐标要求

VLM 输出可接受两种坐标：

```text
bbox_xyxy: 原始图像像素坐标 [x1, y1, x2, y2]
bbox_grid_xyxy: 网格坐标 [gx1, gy1, gx2, gy2]
```

如果 VLM 输出网格坐标，程序再换算回原始图像像素坐标。

## 9. Prompt 设计

### 9.1 系统角色

大模型应被要求扮演：

```text
严谨的分类学图像预标注助手
```

而不是“最终标注专家”。

### 9.2 用户指令重点

Prompt 应包含：

- 当前图像尺寸；
- 网格尺寸；
- 允许输出的部位名称；
- 只输出 JSON；
- 不确定就不要输出；
- 框要宽松覆盖结构主体；
- 不要凭空猜测；
- 输出 confidence 和简短 reason。

### 9.3 输出 schema

建议结构：

```json
{
  "schema_version": "taxamask-vlm-first-mile-v1",
  "detections": [
    {
      "part": "Head",
      "bbox_xyxy": [120, 80, 320, 260],
      "bbox_grid_xyxy": [2.1, 1.4, 5.3, 4.7],
      "confidence": 0.82,
      "reason": "head capsule is clearly visible"
    }
  ]
}
```

## 10. 目标部位策略

### 10.1 用户勾选优先

第一轮不应默认把项目中所有已创建部位都交给 VLM。

推荐规则：

```text
设置页中勾选的 VLM 预标注部位
-> 作为 prompt 中允许输出的目标部位
-> 也是写回项目时允许接受的部位
```

如果用户没有手动勾选目标部位，程序不应直接调用 VLM，而应提示用户先到设置中选择要预标注的部位。

### 10.2 建议候选集

第一轮可以在设置页提供常见蚂蚁外部形态结构作为建议候选，但是否启用由用户勾选决定：

```text
Whole body
Head
Mesosoma
Gaster
Petiole
Postpetiole
Eye
Mandible
Antenna
Scape
Pronotum
Mesonotum
Propodeum
Clypeus
```

这些部位仍必须满足两个条件：

```text
当前项目 taxonomy 中存在
用户在 VLM 预标注设置中勾选
```

### 10.3 与项目 taxonomy 的关系

程序只允许写入当前项目 taxonomy 中存在的部位。

例如：

```text
VLM 输出 Clypeus
但当前项目 taxonomy 没有 Clypeus
-> 拒绝写入
-> 在报告中记录 unknown_or_untargeted_part
```

### 10.4 为什么需要这个限制

它可以避免 VLM 自己发明标签，污染项目结构树。

## 11. 写回项目的规则

### 11.1 当前图像 GUI 写回

建议规则：

```text
VLM box 可写入 auto_boxes，并显示为橙色 aibox
SAM polygon 可写入 parts，但 description 标记为 Auto-Annotated
人工确认后移除 Auto-Annotated 标记
```

每个 AI 草稿建议记录来源信息：

```text
source = vlm_first_mile
source_box_id = 对应 aibox 的 id
source_model = 使用的多模态模型名
source_run_id = 本次预标注运行 id
review_status = draft
```

这不是为了让界面复杂化，而是为了让程序知道某个 polygon 是由哪个 `aibox` 生成的，便于后续确认、替换和清理。

### 11.2 纯框草稿

如果 VLM 给出 box，但 SAM 没有生成可靠 polygon：

```text
只保存 auto_box
不保存 parts polygon
仍保持 Auto-Annotated 草稿状态
```

研究意义：

- 画布上能看到候选框；
- 用户可以手工用 box prompt 或手绘继续修正；
- 不会因为只有框就被训练预检误认为完整标注。

### 11.3 人工确认流程

确认流程既保留当前空格确认习惯，也提供一键通过选项。

单个部位确认行为：

```text
用户选中或浏览到某个 AI 草稿 polygon
-> 视觉检查该部位是否正确
-> 如果正确，按空格确认
-> 程序将该 polygon 的 review_status 改为 confirmed
-> 移除 Auto-Annotated 标记
-> 对应 aibox 保留，并标记为已确认来源框
-> 该 polygon 之后允许进入训练预检
```

当前图像一键通过行为：

```text
用户检查当前图像上的 AI 草稿
-> 对明显错误的部位先重新框选并调用 SAM
-> 剩余草稿如果整体可接受，点击“一键通过当前图像 AI 草稿”
-> 程序把当前图像剩余 Auto-Annotated polygon 批量改为 confirmed
-> 保留每个对应 aibox，并标记为已确认来源框
-> 这些 polygon 之后允许进入训练预检
```

对用户来说，这个动作的含义是：

```text
我已经看过这个草稿，并接受它作为训练标注。
```

一键通过不等于程序自动信任 VLM。它表示研究者已经看过当前图像，认为剩余草稿没有明显问题，可以批量确认。

### 11.4 人工修正流程

如果 AI 框或 SAM polygon 不准确，用户不需要先做复杂删除。

推荐行为：

```text
用户选择当前部位
-> 重新画框
-> 调用 SAM 生成新的 polygon
-> 程序自动移除同一图像、同一部位、同一来源的 aibox
-> 程序自动移除旧的 Auto-Annotated polygon
-> 新 polygon 按人工结果保存
-> 新 polygon 可进入训练预检
```

如果用户只是想先保留 AI 草稿作为参考，也可以先不确认、不替换；未复核草稿仍不会进入训练。

为什么这样设计：

- 用户修正错误时只需要做“正确的动作”，不需要先清理错误草稿；
- 一旦人工重新标注，程序就默认研究者已经作出判断；
- 同一部位不会同时把旧 AI polygon 和新人工 polygon 都送进训练。

### 11.5 与 Clear AI Labels 的关系

`Clear AI Labels` 应能清理：

- `Auto-Annotated` 的 polygon 草稿；
- 对应的 AI 草稿框；
- VLM box-only 草稿。

这样用户可以安全试错。

## 12. 训练预检规则

### 12.1 程序行为

训练预检应跳过仍然标记为 `Auto-Annotated` 的草稿。

只有经过人工审核动作后的标签才进入：

- Locator training；
- SAM/parts training；
- 后续 dataset export 中的训练样本。

允许的审核动作包括：

```text
单个部位空格确认
当前图像一键通过 AI 草稿
人工重新框选并调用 SAM 替换 AI 草稿
```

仍然不能进入训练的内容：

```text
未确认的 Auto-Annotated polygon
只有 aibox、没有对应 confirmed polygon 的候选框
VLM 输出但被拒绝或低置信度过滤的候选
```

### 12.2 为什么重要

VLM 第一公里的价值是降低启动工作量，不是降低数据审查标准。

如果未复核草稿进入训练，后果可能是：

- 错误部位位置被模型学习；
- 小部位专家被错误轨迹污染；
- 后续自动标注越来越自信但越来越偏；
- 研究者难以追溯错误来源。

### 12.3 操作后果

用户需要完成复核动作：

```text
查看草稿
-> 先修正明显错误的 polygon/box
-> 对正确草稿单个空格确认，或对当前图像剩余草稿一键通过
-> 再训练
```

这会增加一次审核动作，但不要求每个部位都逐个点击；重点是训练前有明确的人类判断。

## 13. 批量和无头工具

### 13.1 GUI 批量策略

设置页提供处理范围：

```text
当前图像
已导入的所有图像
```

第一轮建议：

- 默认选择 `当前图像`；
- `已导入的所有图像` 需要用户在设置中主动选择；
- 点击右侧栏 VLM 预标注按钮时，按设置中的处理范围执行；
- 如果处理范围是 `已导入的所有图像`，运行前必须给出二次确认提示；
- 二次确认提示应说明会调用多模态 API、可能产生 API 成本、写入 AI 草稿、保存报告和网格图；
- 批量结果仍全部是草稿，不能绕过人工复核进入训练。

批量执行方式：

```text
图像 1：生成网格图
图像 1：调用 VLM
图像 1：解析结果
图像 1：写入 aibox
图像 1：调用 SAM 生成草稿 polygon
图像 1：保存报告和产物
-> 完成后再进入图像 2
```

也就是说，批量流程应当是线性的：一个图像的一个步骤完成后，再进入下一个步骤或下一张图。不要同时对多张图并发写项目，以减少 API 失败、写回冲突和中断恢复的复杂度。

GUI 进度条：

```text
总步骤数 = 待处理图像数 x 每张图的步骤数
当前进度 = 已完成步骤数 / 总步骤数
```

建议每张图的步骤至少包括：

```text
1. 生成网格图
2. 调用 VLM
3. 解析和过滤候选框
4. 写入橙色 aibox
5. 调用 SAM 生成草稿 polygon
6. 保存报告和产物
```

如果某一步失败，应记录失败原因，继续或暂停的策略由实现阶段再定；但报告中必须能看出失败发生在哪张图、哪个步骤。

为什么重要：

- 当前图像适合试跑模型质量；
- 全部图像适合项目早期快速铺草稿；
- 批量调用可能产生 API 成本和大量草稿，需要避免误触。

### 13.2 为什么仍需要 CLI

GUI 适合逐图复核，但第一公里可能需要先对一批图片生成候选框。

CLI 可以用于：

- 小批量预处理；
- 保存网格图和原始响应；
- 自动化回归测试；
- 以后让 Agent 安全地执行可审计批处理。

### 13.3 建议命令

```text
python tools/agentic/vlm_preannotate_project.py ^
  --project path/to/project.json ^
  --out path/to/project_vlm_draft.json ^
  --images image1.png,image2.png ^
  --parts Head,Mesosoma,Gaster,Eye,Mandible ^
  --api-settings screener_configs/api_runtime_settings.json ^
  --report path/to/vlm_preannotation_report.json
```

### 13.4 输出物

每次运行应输出：

```text
project_vlm_draft.json
vlm_preannotation_report.json
vlm_preannotation/
  run_YYYYMMDD_HHMMSS/
    *_grid_*.png
    *_raw_response_*.txt
    *_vlm_preannotation_*.json
```

含义：

- `project_vlm_draft.json`：带草稿框的新项目文件；
- `vlm_preannotation_report.json`：批量汇总，记录成功、失败、拒绝原因；
- `*_grid_*.png`：实际发给 VLM 的网格图；
- `*_raw_response_*.txt`：模型原始输出，便于追溯；
- `*_vlm_preannotation_*.json`：单图解析结果。

GUI 运行时，报告和网格图建议保存在项目文件同级目录：

```text
项目文件同级目录/
  vlm_preannotation/
    run_YYYYMMDD_HHMMSS/
      report.json
      grid_images/
      raw_responses/
      parsed/
```

这样项目 JSON 保持相对干净，审计材料也能随项目一起保存。

## 14. 安全和审计

### 14.1 不覆盖人工结果

默认规则：

```text
已有人工 polygon 的部位不覆盖
已有人工 box 的部位不覆盖
```

如果未来需要覆盖，必须提供显式选项和确认对话。

### 14.2 报告必须记录拒绝原因

常见拒绝原因：

```text
unknown_or_untargeted_part
below_confidence_threshold
invalid_box
image_not_found
api_error
sam_failed
```

### 14.3 API 原始响应保留

保留 raw response 的目的：

- 判断是模型输出问题还是解析问题；
- 复查某个错误框为什么出现；
- 后续改 prompt 时有对照材料。

注意：

- raw response 不应包含 API key；
- 不应写入根目录；
- 应放在项目目录下的运行产物目录，或 `.tmp_validation/` 中的临时验证目录。

## 15. 与 Blink 工作线的关系

### 15.1 VLM 先于 Blink

VLM 第一公里主要解决：

```text
父部位和常见子部位还没有任何初始框
```

Blink 主要解决：

```text
已有父级上下文后，小部位如何精修、生成 shrink trajectory、训练 route expert
```

### 15.2 推荐组合

```text
VLM 生成 Head / Mesosoma / Gaster / Eye / Mandible 初稿
-> 人工确认 Head 父级框
-> 在主工作台 Blink 面板中选择 Head -> Mandible 或 Head -> Eye
-> 手工修正少量子部位
-> 运行 auto-shrink 生成 trajectory
-> 训练当前子部位专家
```

这样 VLM 负责启动，Blink 负责稳定迭代。

## 16. UI 细节建议

### 16.1 第一轮最小 UI

第一轮建议包含两个轻量改动：

右侧栏上方按钮：

```text
VLM 预标注
```

设置页新增区域：

```text
AI 多模态预标注
```

行为：

- 按设置选择处理当前图像或已导入所有图像；
- 使用默认 `12 x 12` 网格；
- 使用设置中勾选的 VLM 预标注部位；
- 写入草稿层；
- VLM box 显示为橙色 `aibox`；
- SAM polygon 标记为 `Auto-Annotated`；
- 空格确认或一键通过当前图像后才允许进入训练；
- 人工重新框选并调用 SAM 后，自动移除该部位对应 AI 草稿；
- 在日志中显示保存了多少框、多边形、失败项和报告路径。

### 16.2 第二轮可选 UI

如果第一轮稳定，再加：

- 置信度阈值；
- 网格密度；
- 仅生成框，不调用 SAM；
- 打开本次报告目录。

### 16.3 不建议一开始做复杂面板

原因：

- 目前最关键的是验证 VLM 在真实蚂蚁图上的稳定性；
- 过早做复杂 UI 会掩盖核心问题；
- 研究者需要先知道结果是否值得信任。

这里的“不要复杂面板”不是说不能配置，而是把配置集中在设置页，日常入口只负责执行。

## 17. 参数建议

第一轮默认值：

```text
grid_cols = 12
grid_rows = 12
min_confidence = 0.25
max_side = 1600
max_target_parts = 18
temperature = 0
```

解释：

- `12 x 12`：比九宫格细，但还不至于让模型被过密网格干扰；
- `0.25`：第一公里宁可多给候选，让人复核，但不能太低；
- `1600`：兼顾图像细节和 API 传输成本；
- `18` 个目标：避免 prompt 过长，同时覆盖常见蚂蚁结构；
- `temperature = 0`：减少随机性，便于复查。

## 18. 验收标准

### 18.1 最小验收

- 当前图像能生成网格图；
- 设置页能从项目已有部位中勾选 VLM 预标注目标；
- 右侧栏上方按钮能按设置处理当前图像；
- 模型输出能被解析为项目内合法且已勾选的部位；
- 解析出的框显示在画布 `auto_boxes` 层，并呈现为橙色 `aibox`；
- SAM 成功时生成 `Auto-Annotated` polygon；
- SAM 失败时只保留 box 草稿；
- 未复核草稿不会进入训练预检；
- 空格确认后，草稿变成可训练人工确认标注；
- 当前图像一键通过后，剩余 AI 草稿批量变成可训练人工确认标注；
- 确认后的橙色 `aibox` 保留为已确认来源框；
- 用户重新框选当前部位并调用 SAM 后，对应 `aibox` 和旧 AI polygon 会自动移除；
- 一键通过只对当前图像生效；
- 处理已导入所有图像前必须二次确认；
- 批量处理按图像和步骤线性执行；
- GUI 进度条按完成步骤数计算；
- `Clear AI Labels` 能清掉这些草稿；
- 报告记录原始响应、候选框、拒绝原因和产物路径；
- 报告和网格图保存在项目文件同级的 `vlm_preannotation/` 目录中。

### 18.2 研究可用性验收

建议用 10-20 张真实蚂蚁分类学图片试跑，记录：

- Head 框可用率；
- Mesosoma 框可用率；
- Gaster 框可用率；
- Eye 框可用率；
- Mandible 框可用率；
- SAM polygon 可直接接受比例；
- 需要轻微修正比例；
- 完全错误比例；
- 平均每张节省的人工框选时间。

### 18.3 不能通过的情况

如果出现以下情况，应停止批量使用：

- 常见部位错框率很高；
- VLM 经常把比例尺、文字、插图当成结构；
- 多视图图版中经常框错个体或框错视图；
- 输出 JSON 经常不可解析；
- SAM 生成的 polygon 大量飞出框外；
- 研究者复核时间不低于从零手工画框。

## 20. 需要用户确认的问题

仍建议最后确认这些点：

1. 批量处理中某张图或某个步骤失败时，是继续处理下一张图，还是暂停等待用户处理？
2. 进度条中是否把 SAM 失败也计为该步骤已完成但结果为失败？
3. 一键通过当前图像时，是否只确认已有 SAM polygon 的草稿，纯 `aibox` 且没有 polygon 的候选是否仍保持未确认？

已根据本轮讨论暂定的点：

- VLM 结果只作为草稿，不能直接进入训练；
- 复用 PDF Evidence 中现有的 `Multimodal LLM API` 设置；
- 训练预检跳过未复核的 `Auto-Annotated` 草稿；
- GUI 入口放在右侧栏上方，与询问 Agent 同列；
- 设置中增加 `AI 多模态预标注` 区域；
- 从当前项目已有部位中勾选要交给 VLM 的目标部位；
- VLM 预标注部位选择与主部位设置分开；
- VLM 目标部位由用户自行勾选，没有勾选则不运行；
- 设置中选择处理当前图像或已导入所有图像；
- 已导入所有图像运行前必须二次确认；
- 批量流程线性执行，一个图像/步骤完成后再进入下一个；
- GUI 进度条按步骤完成度计算；
- 草稿框显示为橙色 `aibox`；
- SAM 草稿仍需人工复核；
- 正确草稿沿用空格确认；
- 当前图像剩余草稿可一键通过；
- 一键通过只对当前图像生效；
- 空格确认或一键通过后，`aibox` 保留为已确认来源框；
- 错误草稿通过人工重新框选并调用 SAM 来替换；
- 人工替换后自动移除对应 `aibox` 和旧 AI polygon；
- 报告和网格图保存在项目文件同级的 `vlm_preannotation/` 目录。

## 21. 推荐结论

建议实施，但要以“审慎草稿层”方式实施。

推荐第一轮落地范围：

```text
右侧栏上方 VLM 预标注按钮
设置页 AI 多模态预标注区域
用户自行勾选目标部位
当前图像 / 已导入所有图像处理范围
已导入所有图像二次确认
线性批量流程
按步骤完成度显示进度条
复用多模态 API 设置
12 x 12 网格
常见蚂蚁结构候选框
橙色 aibox 草稿框
SAM 草稿 polygon
Auto-Annotated 训练保护
空格确认或当前图像一键通过后才进入训练
确认后的 aibox 保留为已确认来源框
人工重新框选后自动替换 AI 草稿
报告和原始响应留档
```

不建议第一轮开放无人值守的全项目自动写回。即使设置中允许处理已导入所有图像，也应保留二次确认、报告留档和训练保护。

原因：

- 这个能力最有价值的是降低第一批素材制作门槛；
- 但它的错误也最容易在训练早期被放大；
- 小范围、可追溯、人工确认，是最适合分类学研究流程的落地方式。
