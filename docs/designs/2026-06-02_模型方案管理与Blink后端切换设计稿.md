# 模型方案管理与 Blink 后端切换设计稿

日期：2026-06-02

## 目标

当前 TaxaMask 的 2D/STL 标注工作台已经分成“父部位标注”和“子部位标注”两类使用场景，但模型配置仍然偏散：

- 主模型可以在“内置 Locator + SAM”和“外部脚本后端”之间切换。
- Blink 子部位专家训练和推理仍然固定走内置 ViT-B 专家。
- route 只记录父部位、子部位、专家文件等信息，不能明确表达专家模型属于哪种后端。

本设计目标是建立一个可保存、可切换、训练推理一致的模型方案系统，让用户能按研究任务选择方案，而不是靠临时改源码或反复填写命令。

## 设计原则

1. 先服务蚂蚁分类学标注流程。
2. 父部位和子部位分开配置，但在同一个模型方案系统里管理。
3. 每个模型方案必须同时说明训练入口、推理入口、权重/manifest 保存方式和输出解释方式。
4. Blink route 必须记录专家后端类型，避免把热力图专家误当作 ViT-B 专家加载。
5. 默认方案必须保持当前可用流程不变。
6. 外部脚本能力保留，但不把所有事情都塞进一个“通用外部后端”输入框。

## 后端类型

### 父部位后端

| 后端 | 用途 | 当前状态 |
|---|---|---|
| `builtin_locator_sam` | 内置热力图 Locator + SAM | 已有 |
| `external_parent` | 外部主模型脚本 | 部分已有 |

### 子部位后端

| 后端 | 用途 | 当前状态 |
|---|---|---|
| `vit_b_blink` | 当前 ViT-B Blink 专家 | 已有但未显式标记 |

| `external_blink` | 外部 Blink 专家脚本 | 待实现 |

## 训练链路设计

### 父部位训练

父部位训练继续走现有主流程：

```text
项目图片 + 标注
  -> training preflight
  -> Locator 数据集
  -> Locator 训练
  -> SAM/分割训练，可选
  -> 保存 locator/segmenter 权重
  -> 更新当前父部位模型方案
```

### ViT-B Blink 训练

当前流程保留，但训练结果必须写 manifest：

```text
子部位 polygon + loose shrink box + 父级框
  -> BlinkRefiner 生成收缩轨迹
  -> BlinkTrajectoryDataset
  -> ViT-B 框回归专家
  -> 保存 .pth + manifest
  -> route 记录 expert_backend=vit_b_blink
```

### 热力图 Blink 训练

新增训练器，复用 Blink 数据集思想，但把目标框转换成单类 heatmap + WH：

```text
BlinkTrajectoryDataset
  -> 父级 ROI 图像
  -> inside/outside 图像
  -> target_step / target_final box
  -> 生成单类 heatmap 中心点
  -> 生成 WH target
  -> 热力图专家训练
  -> 保存 .pth + manifest
  -> route 记录 expert_backend=heatmap_blink
```

## 推理链路设计

### 自动标注当前 / 批量标注全部

```text
图片
  -> 父部位后端推理
  -> 得到父部位 auto_boxes / polygons
  -> 遍历可用 route
  -> 根据 route.expert_backend 分流
      vit_b_blink: ViT-B 专家推理
      heatmap_blink: 热力图专家推理
      external_blink: 外部 Blink 脚本推理
  -> 子部位 box
  -> SAM 生成子部位 polygon
  -> 写入 AI 草稿
```

### 用已有父框标注子部位

```text
当前选中子部位
  -> 找 parent_part
  -> 读取当前图已有父级框
      优先人工框
      其次自动框
  -> 读取 route
  -> 根据 expert_backend 分流
  -> 子部位 box
  -> SAM polygon
  -> 写入当前图
```

这个按钮不负责重新跑父部位自动标注。若无父级框，继续禁用。

## 外部后端契约扩展

现有 `external_backend_contract_v1` 适合主模型。Blink 外部后端应单独定义契约，因为它不是整图推理，而是“父级 ROI 内找子部位”。

建议新增：

```text
docs/contracts/external_blink_backend_contract_v1.md
```

推理 contract 关键字段：

```json
{
  "schema_version": "taxamask_external_blink_contract_v1",
  "action": "predict_child",
  "image_path": "C:/path/image.jpg",
  "parent_part": "Head",
  "child_part": "Mandible",
  "parent_box": [10, 20, 300, 260],
  "prediction_json": "C:/run/prediction.json",
  "model_manifest": "C:/model/manifest.json"
}
```

输出：

```json
{
  "schema_version": "taxamask_blink_prediction_v1",
  "child_part": "Mandible",
  "box": [80, 120, 130, 170],
  "score": 0.86,
  "model_id": "external_blink/mandible_v1"
}
```

## 迁移策略

1. 旧项目没有 `model_profiles` 时，自动创建默认方案。
2. 旧 route 没有 `expert_backend` 时，默认解释为 `vit_b_blink`。
3. 旧 `.pth` 专家没有 manifest 时，继续按 ViT-B legacy 加载，但在 UI 中显示“旧 ViT-B 专家”。
4. 不自动覆盖旧 route，用户确认后才写入新 manifest。

## 分阶段实施

### 阶段 1：方案结构与 UI 框架

- 新增模型方案数据结构。
- 设置页增加“方案 / 父部位标注 / 子部位标注”布局。
- route 增加 `expert_backend` 字段。
- 保持当前 ViT-B 行为不变。

### 阶段 2：Blink 后端注册表

- `CascadingManager` 改为根据 `expert_backend` 加载专家。
- ViT-B 专家作为第一个注册后端。
- 旧专家兼容。

### 阶段 3：热力图 Blink 专家

- 实现热力图 Blink 数据集和训练器。
- 新增热力图专家推理。
- route 可选择热力图专家。

### 阶段 4：外部 Blink 契约

- 定义外部 Blink contract。
- 支持外部 Blink 训练/推理。
- UI 支持保存外部 Blink 方案。

### 阶段 5：方案模板与审计

- 增加方案复制、导入、导出。
- 训练/推理日志记录当前方案 ID。
- 输出报告记录方案快照，方便论文和复现实验。

## 风险与取舍

### 风险

- 如果 route 不记录 `expert_backend`，热力图专家会被误当 ViT-B 加载。
- 如果训练和推理读取不同方案，用户会以为训练成功但推理失败。
- 外部后端过早做成大而全，可能增加配置复杂度。

### 取舍

- 先做内置方案化和 route 后端分流，再做外部 Blink。
- 保留 ViT-B，不直接删除。
- 热力图 Blink 作为推荐默认候选，但等测试后再设为默认。
