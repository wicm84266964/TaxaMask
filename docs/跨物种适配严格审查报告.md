# 跨物种适配严格审查报告

> 日期：2026-05-03
>
> 审查对象：TaxaMask / Formica-Flow 当前主项目
>
> 审查立场：按准备开源上传 GitHub 的标准，检查它是否真的适合宣称“可用于多种生物分类学掩码标注和数据集产出”。

---

## 总体结论

当前项目可以谨慎宣称：

> 支持通过项目级 taxonomy、locator scope、筛选 profile 和导出格式适配不同生物分类学图像任务。

但不建议直接宣称：

> 已经完整支持所有生物类群，或已经在多类群上充分验证。

更稳妥的产品定位是：

> 从蚂蚁分类学工作流长出来的通用化生物掩码标注与数据集生成工具；当前蚂蚁场景验证最充分，其他类群需要项目级配置和人工复核。

这对研究任务的意义是：开源后不会误导别人把默认蚂蚁部位、蚂蚁文献筛选词和蚂蚁级联路线直接套到植物、甲虫、化石或显微图像上。

---

## 已经做得比较好的地方

### 1. 项目 taxonomy 不是固定写死

程序在 `AntSleap/core/project.py` 中保存项目自己的 `taxonomy`，并通过 `sanitize_taxonomy(...)` 清理非法标签。`AntSleap/core/engine.py` 中的 Locator 会根据 `num_classes` 重建输出通道。

它在做什么：允许一个项目拥有自己的标注标签集合。

为什么重要：不同类群的“部位”不是同一套词。蚂蚁可以是 Head/Mesosoma/Gaster，植物可能是 Leaf/Flower/Fruit，甲虫可能是 Head/Pronotum/Elytra。

代价或后果：模型权重和项目 taxonomy 强绑定。换 taxonomy 后不能假设旧 Locator 权重还能直接可信使用。

### 2. locator_scope 与完整 taxonomy 分开

当前项目可以把大部位定位范围 `locator_scope` 和完整标注标签 `taxonomy` 分离。

它在做什么：第一阶段只找适合粗定位的大结构，第二阶段可以标注更细标签。

为什么重要：跨物种时，很多细结构不适合在整图中直接定位。先找叶片、头部、翅鞘、整体器官，再做局部细分更符合真实标注工作。

代价或后果：用户必须理解“哪些标签适合做 Locator，哪些只适合做局部 SAM/Blink 标注”。这需要 README 和手册继续加强。

### 3. PDF 筛选逻辑已经 profile 化

`screener_configs/*.json` 可以配置 required/supportive/exclude/taxonomic group keywords。

它在做什么：文献筛选不必永远只筛蚂蚁，可以换成其他分类群关键词。

为什么重要：开源用户搜索和筛选的文献对象可能是甲虫、植物、鱼类、化石或寄生蜂。

代价或后果：默认 `默认Legacy方案.json` 仍然强烈偏蚂蚁。必须提供更多示例 profile，否则其他类群用户会误以为程序“能换类群”，但不知道应该改哪里。

### 4. 数据导出保留来源追踪

`image_provenance` 和 multimodal JSONL 的 `source_provenance` 能记录 PDF 候选来源。

它在做什么：候选图从哪篇 PDF、哪一页、哪个候选流程来，可以跟随数据导出。

为什么重要：跨研究团队整合数据集时，最怕的是来源不清、复核状态不清、训练样本混入不可追溯材料。

代价或后果：目前 provenance 记录的是工程来源，不等于完整的生物学 metadata。物种名、标本号、采集地、模式状态等还需要更正式的 schema。

## 主要不足和风险

### 1. 默认样例仍然强绑定蚂蚁形态

`AntSleap/core/taxonomy_defaults.py` 默认是 `Head`, `Mesosoma`, `Gaster`。模板项目也仍然以 Formica/ant 部位作为示例。

它在做什么：新项目默认进入蚂蚁身体部位工作流。

为什么影响研究流程：非蚂蚁用户第一次打开项目时，可能会直接沿用错误标签开始标注，导致后续训练集语义混乱。

建议：保留蚂蚁模板，但新增更中性的 `generic_specimen` 模板，例如 `Object`, `Region`, `Structure`，并在新建项目时让用户选择模板。

### 2. 文献筛选默认 profile 对其他类群偏差很大

默认 supportive keywords 包含大量蚂蚁形态词，如 mesosoma、petiole、gaster、scape 等。

它在做什么：帮助蚂蚁分类学 PDF 筛选更准。

为什么影响研究流程：换成植物或甲虫时，这些词会降低召回率或引入错误评分，让候选文献选择偏向蚂蚁论文。

建议：至少提供 `ants.example.json`、`generic_taxonomy.example.json`、`plants.example.json`、`insects_generic.example.json` 四类 profile。

### 3. Blink 级联语义目前是“父部位到子部位”，但缺少类群模板

项目可以保存 `cascade_routes` 和 `blink_context_roi_parents`，但默认经验来自 `Head -> Mandible`、`Head -> Eye`。

它在做什么：允许在大部位内训练或应用子部位专家。

为什么影响研究流程：跨物种时，父子关系并不通用。植物、鱼类、化石板图、显微切片都可能需要完全不同的层级。

建议：把 route 解释成“局部上下文关系”，不要在文档中暗示它只适合解剖父子关系；同时提供项目模板中的 route 示例。

### 4. 路径恢复仍含 Formica-Flow_output 特例

`ProjectManager._resolve_known_relocated_output(...)` 对 `Formica-Flow_output` 和 `C:\savedata\Formica-Flow_output` 有历史兼容逻辑。

它在做什么：帮助作者本机旧数据搬盘后恢复路径。

为什么影响开源：这是个人/历史工作流痕迹。它不一定伤害运行，但会让外部开发者觉得路径策略不够通用。

建议：后续改成可配置的 `known_relocated_roots`，不要把特定输出目录写入代码。

### 5. “物种适配”目前更多是标签适配，不是完整分类学 metadata 适配

程序支持项目标签、图像来源、标注导出，但还没有完整的 Darwin Core 或自然历史标本 metadata schema。

它在做什么：解决掩码标注和训练数据产出。

为什么影响长期愿景：如果目标是汇聚大家的数据集并推动分类学模型能力提升，只有 mask 不够，还需要物种名、鉴定状态、标本号、类型状态、出处、许可证、复核人等字段。

建议：把下一阶段数据 schema 分成两层：

- annotation schema：图像、mask、标签、box、polygon、质量状态
- taxonomy/specimen schema：taxon name、identifier、source、license、review status、voucher/specimen metadata

### 6. 权重和训练输出曾经直接混在主目录

当前 `.gitignore` 已经阻止权重、实验输出、数据库和 artifacts 上传，但现有工作区仍有大量本地文件。

它在做什么：本地研究迭代留下了真实训练权重、数据库、实验图、测试项目 JSON。

为什么影响开源：如果手动拖拽上传或忽略 `.gitignore`，会把大文件、个人路径、甚至敏感配置带到 GitHub。

建议：真正初始化 Git 前，用 `git status --ignored` 或等价清单复查；不要用“全选上传”。

## 建议的开源表述

推荐：

> TaxaMask is an open-source tool for biological mask annotation, specimen image segmentation, and dataset generation for taxonomy research. It is ant-tested first and designed to be adapted to other taxa through project-level taxonomies and screening profiles.

不推荐：

> TaxaMask supports all taxa out of the box.

中文推荐：

> TaxaMask 是一个面向分类学研究的生物掩码标注与数据集生成工具。它从蚂蚁分类学工作流出发，当前蚂蚁场景验证最充分，并通过项目级 taxonomy、locator scope 和筛选 profile 支持向其他分类群扩展。

---

## 优先改进清单

1. 新增通用项目模板，避免非蚂蚁用户默认继承 Head/Mesosoma/Gaster。
2. 新增 3-4 个非蚂蚁筛选 profile 示例。
3. 把 `Formica-Flow_output` 路径恢复特例改成可配置 root remap。
4. 为导出的 multimodal dataset 增加更正式的 specimen/taxon metadata 占位字段。
5. 在 GUI 新建项目流程中加入 taxonomy 模板选择。
6. 增加一个最小非蚂蚁测试项目，验证自定义 taxonomy、训练预检、导出和 CLI 不依赖蚂蚁标签。

---

## 审查结论

当前代码已经具备跨物种适配的基础接口，但“跨物种产品化”还没有完全完成。开源发布时最稳的策略是：

- 名字和 README 面向更广义的 taxonomy mask annotation；
- 默认叙述明确“starting from ants”；
- 不把蚂蚁验证经验包装成全类群验证；
- 把其他类群能力描述为可配置、可扩展、需要复核的工作流。

这样既不压低项目愿景，也不会牺牲研究数据的可信度。
