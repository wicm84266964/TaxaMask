# AI 助手交接文档：AntSleap 项目修复说明

**日期**: 2025年12月25日
**交接人**: Gemini Pro

## 1. 核心问题背景

用户反馈了两个主要问题：
1.  **推理精度问题**：在使用一个仅包含“头部”标注训练的模型进行自动标注时，标注结果从“精准定位头部”退化为“框住整只蚂蚁”或“在全图标注大量混乱的点”。
2.  **分类体系管理问题**：
    *   在 UI 上通过“+”/“-”按钮增删分类部位（Taxonomy Part）后，操作无法被持久化保存。
    *   加载一个仅包含一个自定义分类部位的项目文件时，程序依然显示默认的五个分类部位。

## 2. 问题分析与修复尝试（包含失败的过程）

### 2.1 推理精度问题 (框体过大)

#### 初步诊断
这个问题的根源在于 `core/engine.py` 的 `predict_full_pipeline` 方法中，用于从热力图（Heatmap）计算包围框（Bounding Box）的**阈值参数过于宽松**。

其逻辑如下：
1.  模型输出一张热力图。
2.  程序找到热力图中的最大值 `max_val`。
3.  一个固定的**置信度阈值 (Confidence Threshold)**（默认为 `0.1`）决定了是否继续处理。
4.  如果继续，一个**自适应阈值** `threshold = max_val * 0.4` 被用来圈定有效像素点。
5.  所有高于 `threshold` 的像素点被用来计算最终的包围框。

当模型过拟合或不确定时，`max_val` 会很低（例如 `0.2`），导致 `threshold` 变得极低（例如 `0.08`）。这个极低的 `threshold` 会把图像上大量的背景噪音都识别为有效像素，从而计算出一个覆盖整张图的巨大包围框，最终让 SAM 模型标注了整只蚂蚁。

#### 修复尝试与失败
我的目标是将这些固定的阈值参数暴露到 UI，让用户可以动态调整。

1.  **暴露参数到 UI**：
    *   我修改了 `core/engine.py`，使 `predict_full_pipeline` 方法可以接收 `conf_thresh` 等参数。
    *   我修改了 `core/config.py`，增加了这些参数的默认值。
    *   我修改了 `main.py`，创建了 `ModelSettingsDialog` 对话框，并将其与菜单和配置系统连接起来。

2.  **引入“底噪阈值” (Noise Floor)**：
    *   在与用户的交流中，我意识到仅有比例阈值是不够的，还需要一个绝对的最低门槛。
    *   因此，我将 `core/engine.py` 中的逻辑修改为 `threshold = max(max_val * adapt_thresh, noise_floor)`，并同样将其暴露到了 UI 中。

3.  **失败点**：在反复修改 `main.py` 和 `core/engine.py` 的过程中，由于我使用的 `replace` 和 `run_shell_command` 工具在处理大文件和特殊字符时出现严重问题，**导致这两个核心文件被反复损坏**。具体的损坏形式包括：
    *   **字符串编码错误**：中文字符被替换为 `?` 或 ``。
    *   **语法错误**：出现未闭合的字符串、多余的括号等。
    *   **缩进错误**：最致命的是，`predict_full_pipeline` 方法的定义因缩进错误而脱离了 `AntEngine` 类，导致了最终的 `AttributeError: 'AntEngine' object has no attribute 'predict_full_pipeline'` 报错。

尽管我最后试图用一个完整的、我认为是正确的版本去覆盖 `core/engine.py`，但这个操作也失败了，未能解决 `AttributeError`。这表明该文件仍处于损坏状态。

### 2.2 分类体系管理问题

#### 初步诊断

1.  **无法保存增删操作**：这是因为 `core/project.py` 中的 `add_taxonomy_part` 和 `remove_taxonomy_part` 方法在执行后会立刻保存，而 `main.py` 在调用它们之后又会调用 `refresh_ui`，这其中可能存在状态不一致的问题。
2.  **加载项目时分类不正确**：`core/project.py` 的 `load_project` 方法中，存在一个逻辑缺陷：当加载的项目文件 `taxonomy` 列表为空时，它会错误地退回到使用程序默认的五个分类，而不是保持为空。

#### 修复尝试
1.  **修正加载逻辑**：我修改了 `load_project` 方法，使其严格使用文件中的 `taxonomy` 列表，即便是空列表也直接加载。
2.  **调整保存时机**：我移除了 `add_taxonomy_part` 和 `remove_taxonomy_part` 方法中的 `save_project()` 调用，并将其移动到 `main.py` 中，在确认 UI 刷新成功后再进行保存。

**这些针对 `core/project.py` 的修改应该是正确且有效的**。问题主要出在 `main.py` 和 `core/engine.py` 的反复损坏上。
