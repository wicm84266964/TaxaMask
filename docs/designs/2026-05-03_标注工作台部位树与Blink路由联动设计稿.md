# 标注工作台部位树与 Blink 路由联动设计稿

日期：2026-05-03

## 1. 目标

当前标注工作台的部位列表是平铺纵向列表。随着项目支持更多部位，尤其是进入 Blink 后形成 `父部位 -> 子部位` 关系，单列列表会越来越难查找和管理。

本设计目标是在不重构项目底层数据格式的前提下，把标注工作台的部位管理区改成可折叠树形展示：

- 大部位仍作为主要入口。
- 明确经过 Blink 建立父子关系的子部位归到父部位下。
- 跨区域结构不强行归入某个父部位。
- 未建立父子关系的新增部位放到未分组区域。
- 删除和添加仍走现有项目管理逻辑，确保标签、轨迹、路由、专家引用不会残留。

这次改动本质上是“展示与交互层优化”，不是项目数据结构重写。

## 2. 不做事项

第一版不做以下内容：

- 不把 `taxonomy` 从平铺列表改成树。
- 不把 `cascade_routes` 改成新的树形 JSON。
- 不在标注工作台第一版里做拖拽改父部位。
- 不在标注工作台里直接编辑复杂路由参数。
- 不把跨区域结构强行放到任意父部位下。
- 不改变训练、导出、推理读取 taxonomy 的方式。

这些限制是为了保护已有项目文件和训练链路。

## 3. 现有数据边界

### 3.1 taxonomy

`taxonomy` 是项目内所有可标注部位的平铺列表。

示例：

```text
Head
Mesosoma
Gaster
Mandible
Eye
Setae
```

它决定：

- 标注工作台可选择哪些部位。
- SAM / parts 阶段可以训练哪些掩码。
- 导出数据集包含哪些类别。
- 推理输出会按 taxonomy 过滤。

### 3.2 locator_scope

`locator_scope` 是主工作台宏定位器训练和推理的直接定位范围。

蚂蚁模板默认：

```text
Head
Mesosoma
Gaster
```

普通 UI 新增部位不会自动加入 `locator_scope`。

### 3.3 cascade_routes

`cascade_routes` 保存 Blink 父子关系和专家绑定。

示例：

```text
Head -> Mandible
Head -> Eye
Mesosoma -> Leg
```

路由决定标注工作台一键推理时是否可以执行：

```text
宏定位器找到父部位 -> Blink 微专家找子部位 -> SAM 生成子部位掩码
```

### 3.4 blink_context_roi_parents

`blink_context_roi_parents` 记录某个子部位最近使用过的父部位 ROI。

它用于下一次进入 Blink 时优先推荐之前用过的父部位。

## 4. 展示规则

树形 UI 只根据现有平铺数据生成，不写入新的树结构。

### 4.1 顶层主部位

优先使用 `locator_scope` 里的部位作为顶层主部位。

蚂蚁项目示例：

```text
Head
Mesosoma
Gaster
```

如果某个父部位不在 `locator_scope`，但存在有效子路由，也可以作为顶层父节点显示，避免已有 Blink 关系在 UI 中丢失。

### 4.2 单父部位子结构

如果某个部位只作为一个父部位的子部位出现，则显示到该父部位下。

示例：

```text
Head
  Mandible
  Eye
```

判断依据：

- `cascade_routes` 中存在 `Head -> Mandible`
- `Mandible` 没有其他父部位路由

### 4.3 跨区域结构

如果同一个部位出现在多个父部位路由中，不自动归入任何一个父部位，显示到“跨区域结构”。

示例：

```text
跨区域结构
  Setae
  Spiracle
  Punctures
```

这类结构可能分布在全身或多个区域，强行归到单一父部位会误导用户。

第一版规则：

- 子部位有 2 个或以上不同父部位路由时，归入“跨区域结构”。
- 不根据名字自动判断刚毛、气孔、刻点等结构。
- 不重复显示同一个跨区域结构。

### 4.4 未分组部位

不在 `locator_scope`、也没有 Blink 父子路由的部位显示到“未分组部位”。

示例：

```text
未分组部位
  Wing
  Unknown structure
```

新增部位默认进入这里，直到用户通过 Blink 建立明确父子关系。

### 4.5 重名和安全性

继续沿用现有 `is_safe_part_name` 规则：

- 不允许空名称。
- 不允许路径分隔符。
- 不允许 `..`、绝对路径、冒号等危险字符。
- 不允许与现有 taxonomy 重名。

## 5. UI 设计

### 5.1 控件替换

将标注工作台当前 `QListWidget part_list` 替换或包裹为树形控件，建议使用 `QTreeWidget`。

建议列：

```text
部位
状态
```

第一列显示部位名称，第二列显示简短状态。

### 5.2 状态显示

子部位状态建议：

- `Blink 已启用`：路由已启用且有指定专家。
- `有候选专家`：存在专家文件或候选，但路由未启用或未指定。
- `未训练`：有父子路由，但没有可用专家。
- 空白：普通主部位或未分组部位。

状态只作为辅助显示，不影响点击选择。

### 5.3 选择行为

点击任何可标注部位节点，都应等价于当前列表选择部位：

- 设置画布当前部位。
- 更新描述框。
- 如果启用形态测量，刷新测量。
- 保持现有标注工具行为。

分组节点不可作为标注目标：

- “跨区域结构”
- “未分组部位”

### 5.4 折叠行为

默认展开：

- 当前选中部位所在父节点。
- 有启用 Blink 路由的父节点。

默认折叠：

- 跨区域结构可按部位数量决定；数量少时展开，数量多时折叠。
- 未分组部位建议默认展开，方便找到新建部位。

### 5.5 搜索过滤

建议增加一个小搜索框。

搜索时：

- 匹配部位名称。
- 展开匹配节点的父节点。
- 隐藏不相关叶子节点。
- 分组节点如果有匹配子节点则保留。

第一版可以先做大小写不敏感的包含匹配。

## 6. 添加部位流程

添加部位继续调用：

```text
ProjectManager.add_taxonomy_part(part_name)
```

流程：

1. 用户点击添加。
2. 输入部位名称。
3. 调用 `add_taxonomy_part`。
4. 成功后刷新树。
5. 新部位显示到“未分组部位”。
6. 自动选中新部位。

不做：

- 不自动创建父子路由。
- 不自动加入 `locator_scope`。
- 不自动绑定 Blink 专家。

理由：

用户添加 `Mandible` 只是说明项目有这个可标注类别，并不等于已经确认它应该由 `Head` 接力定位。

## 7. 删除部位流程

删除部位必须继续调用：

```text
ProjectManager.remove_taxonomy_part(part_name)
```

禁止只删除 UI 节点。

现有删除逻辑会清理：

- taxonomy 中的部位。
- locator_scope 中相关项。
- 所有图片里的该部位标签。
- 该部位的 manual boxes。
- 该部位的 auto boxes。
- 该部位的 descriptions。
- 该部位的 Blink trajectories。
- 相关 `blink_context_roi_parents`。
- 相关 `cascade_routes`。

删除前确认弹窗应说明：

```text
删除该部位会同时删除所有图片中的该部位标注、Blink 轨迹，以及涉及该部位的父子路由。此操作会影响训练和导出结果。
```

第一版不自动删除磁盘上的专家权重文件。

原因：

- 专家权重可能被用户用于备份、比较或其他项目。
- 当前已有专家桶删除流程带高风险确认，不应被普通删除部位静默触发。

删除后：

- 刷新树。
- 刷新画布 polygons / boxes。
- 刷新路由管理面板。
- 刷新 Blink 工作台 taxonomy。
- 如果当前选择的是被删除部位，自动选择下一个可用部位。

## 8. Blink 联动流程

### 8.1 进入 Blink 时

用户从标注工作台选择子部位并进入 Blink。

Blink 入口选择父部位 ROI 后：

```text
remember_blink_context_parent(child, parent)
register_cascade_route_candidate(parent, child)
```

然后：

- 标注工作台部位树刷新。
- 模型设置路由面板刷新。
- 子部位从“未分组部位”移动到父部位下。

### 8.2 Blink 训练成功时

如果当前路由没有指定专家：

- 自动绑定新专家。
- 自动启用路由。
- 刷新部位树状态。
- 刷新路由管理面板。

如果当前路由已有指定专家：

- 不覆盖。
- 新模型保存为候选版本。
- 刷新部位树状态。
- 用户可在路由管理中自行切换。

### 8.3 Blink 删除专家桶时

现有专家桶删除逻辑会影响路由。

删除成功后应触发：

- 部位树状态刷新。
- 路由管理面板刷新。
- 当前画布不应丢失已有人工标注。

## 9. 路由管理面板联动

模型设置中的路由管理面板仍然是路由专家绑定的权威入口。

标注工作台树只展示摘要状态，不在第一版中承担复杂路由编辑。

当用户在路由管理面板中：

- 指定专家
- 启用路由
- 停用路由
- 删除路由

都应触发标注工作台树刷新。

## 10. 推理影响

这次 UI 树改造不改变推理算法。

推理仍然按现有逻辑：

1. 主定位器处理 `locator_scope`。
2. 如果项目路由启用且专家可用，按 `父部位 -> 子部位` 调用 Blink 专家。
3. 使用 SAM 生成掩码。
4. 写回项目标签。

树形 UI 只让用户更清楚地看到哪些子部位已经具备接力推理能力。

## 11. 实施清单

### 阶段 A：只读树构建

- 新增一个纯函数或小型 helper，用 `taxonomy`、`locator_scope`、`cascade_routes` 构建展示树模型。
- 覆盖单父部位、跨区域结构、未分组部位。
- 不改 UI。
- 添加单元测试。

验收标准：

- 输入平铺数据，输出稳定树结构。
- 同一子部位多父路由时只进入跨区域结构。
- 无路由子部位进入未分组部位。

### 阶段 B：替换标注工作台部位列表展示

- 将当前部位列表替换为树形控件。
- 点击叶子节点仍调用现有 `on_part_selected` 等价逻辑。
- 分组节点不可选为标注目标。
- 保持添加、删除按钮位置。

验收标准：

- 选择部位后画布当前部位正确变化。
- 标注、SAM 框选、手工绘制不受影响。
- 形态测量刷新不受影响。

### 阶段 C：添加搜索与折叠状态

- 添加部位搜索框。
- 支持按名称过滤。
- 保留当前选中节点展开。
- Blink 路由启用的父部位默认展开。

验收标准：

- 大量部位下可以快速定位目标。
- 搜索不会改变当前项目数据。

### 阶段 D：添加部位联动

- 新增部位后刷新树。
- 新部位默认出现在未分组部位。
- 自动选中新部位。

验收标准：

- 新增部位进入 taxonomy。
- 不自动创建路由。
- 不自动加入 locator_scope。

### 阶段 E：删除部位联动

- 删除仍调用 `remove_taxonomy_part`。
- 删除确认文案说明标注、轨迹、路由会被清理。
- 删除后刷新画布、树、Blink、路由面板。

验收标准：

- 删除父部位时，相关子路由被清理。
- 删除子部位时，相关路由和轨迹被清理。
- 未删除其他部位标注。
- 不静默删除专家权重文件。

### 阶段 F：Blink 路由刷新联动

- 进入 Blink 建立路由后刷新树。
- Blink 训练成功后刷新树状态。
- 专家桶删除后刷新树状态。

验收标准：

- `Head -> Mandible` 建立后，Mandible 从未分组移动到 Head 下。
- 路由启用后状态显示 Blink 已启用。
- 已有专家不被新训练覆盖。

### 阶段 G：模型设置路由管理联动

- 路由面板启用、停用、指定专家、删除路由后触发树刷新。

验收标准：

- 路由面板变更后，标注工作台树状态同步。

### 阶段 H：回归测试

- 标注工作台添加/删除部位。
- 手工画多边形。
- SAM 框选。
- 训练预检。
- Blink 建立路由。
- Blink 训练后自动绑定空路由。
- 已有路由专家不覆盖。
- 删除部位清理路由。
- 数据集导出。

## 12. 风险清单

### 高风险：删除逻辑绕开 ProjectManager

如果 UI 树节点删除直接删界面，会留下标签、轨迹或路由残留。

控制措施：

- 删除只能调用 `remove_taxonomy_part`。
- 测试覆盖删除父部位、删除子部位、删除跨区域结构。

### 中风险：树展示和真实路由不一致

如果路由面板变更后树不刷新，用户会误判当前自动标注状态。

控制措施：

- 路由相关操作统一发出刷新信号。
- 标注工作台订阅该信号刷新树。

### 中风险：跨区域结构误归类

刚毛、气孔等跨区域结构不应被塞进某个父部位。

控制措施：

- 多父路由结构进入“跨区域结构”。
- 第一版不做自动语义分类。

### 中风险：用户以为树结构改变训练类别

树只是展示，不改变 taxonomy、locator_scope、训练逻辑。

控制措施：

- UI 文案避免暗示树会改变训练层级。
- 使用手册中说明“树由 Blink 路由生成，仅用于管理和选择部位”。

## 13. 建议验收顺序

1. 先验收只读树构建规则。
2. 再验收标注工作台树展示。
3. 再验收添加、删除部位。
4. 再验收 Blink 路由自动移动节点。
5. 最后验收训练、推理、导出回归。

不要先做复杂编辑能力。

## 14. 最终判断

这项改动值得做，但必须保持“数据平铺，界面树形”的边界。

父子关系树应由 Blink 路由自然生成，而不是在标注工作台里强迫用户提前建复杂层级。

删除和添加是本功能最大的风险点，必须复用现有项目管理器，不能写第二套清理逻辑。
