# TaxaMask 1.2.0 TIF Developer Preview

Release date: 2026-06-27

Branch: `codex/antscan-stl-tif-rearchitecture`

Tag: `v1.2.0-tif-preview`

This release is the full developer-preview branch release for TaxaMask 1.2.0. It includes the 2D SQLite storage upgrade from the stable 1.2.0 line and also publishes the TIF/CT workflow changes that are intentionally kept on this preview branch.

The stable `v1.2.0` tag remains the 2D-focused release. This preview uses a separate tag so the TIF branch can be downloaded and cited without overwriting the stable release tag.

## Research Workflow Impact

The main goal of this release is data safety for large projects.

Earlier 2D projects used a single large JSON file as the main project record. That made large VLM pre-annotation and review sessions vulnerable: one interrupted full-file write could make the whole project hard to reopen. From this release onward, new 2D projects use SQLite as the primary project storage, and old 2D JSON projects can be migrated automatically when opened.

The TIF/CT route also uses SQLite for the project index while keeping large volume and label sidecar files on disk. This keeps TIF data practical for large specimens without putting heavy arrays into the database.

## Storage And Migration

- New 2D projects are created as SQLite-backed projects.
- New TIF projects are created as SQLite-backed project indexes.
- Opening an old 2D JSON project prompts for migration to SQLite.
- Opening an old TIF JSON project prompts for migration to SQLite.
- Migration creates a project entry file, database, migration report, and legacy JSON backup beside the old project.
- The old JSON source is not overwritten during migration.
- Existing migrated projects are reused when the old JSON is selected again.
- Project opening now makes the entry file clearer:
  - 2D entry: `*.sqlite_manifest.json`
  - TIF entry: `*.tif_sqlite_manifest.json`
- If a user accidentally selects the SQLite database file itself, TaxaMask tries to locate and open the matching project entry file automatically.

## 2D Route

- 2D project loading and saving now use SQLite transactions for the main project record.
- Legacy JSON writing is no longer the normal save path for migrated or newly created projects.
- VLM pre-annotation results, auto boxes, manual polygons, trajectories, scale records, image groups, model profiles, and route metadata are migrated into SQLite.
- SQLite maintenance actions are available for backup and legacy JSON export for audit/debugging.
- Startup placeholder projects now reuse an existing SQLite project entry instead of failing when the manifest already exists.

For researchers, this means a large 2D review project should no longer rely on repeatedly rewriting one huge JSON file after each save.

## TIF / CT Route

- TIF project records can be stored in SQLite while volume data, label volumes, part volumes, masks, and reslice outputs remain as sidecar files.
- Existing TIF JSON projects can be migrated with a progress dialog.
- TIF workbench loading, saving, specimen records, part records, local-axis reslice records, and backend handoff metadata are adapted to the SQLite project index.
- The TIF workbench keeps source TIFF stacks and generated sidecars outside the database.
- Saved local-axis reslice items remain read-only review records and do not modify original part volumes.

For researchers, this means TIF project navigation and metadata are safer to save incrementally, while large CT/TIF arrays stay in normal project folders.

## TIF Annotation Tooling

The TIF annotation page has been changed from an engineering-style paint entry into a clearer annotation tool surface.

- Visible tools:
  - Brush
  - Eraser
  - Picker
  - Pan
- Erasing no longer depends only on a hidden `Ctrl` gesture.
- Picker samples the label under the cursor and switches the current material.
- Brush and eraser show a canvas radius preview.
- Read-only states explain why painting is blocked.
- The material panel shows color swatches and current material feedback.
- Material editing uses a color picker instead of requiring users to type color codes.
- Save status shows whether there are dirty slices and whether the current working edit has been saved.
- Undo/redo buttons and shortcuts are synchronized with annotation history.
- Shortcuts:
  - `B`: Brush
  - `E`: Eraser
  - `I`: Picker
  - `Ctrl+Z`: Undo
  - `Ctrl+Y` / `Ctrl+Shift+Z`: Redo
  - `Ctrl+S`: Save
  - `[` / `]`: smaller/larger brush

Advanced tools such as lasso fill, bucket fill, rectangle/ellipse fill, cross-slice propagation, and smoothing are intentionally left as design-level placeholders. Each needs separate undo and risk handling before it is exposed in the UI.

## Validation

Validation performed during this release cycle included:

- 2D SQLite schema, migration, load, save, and maintenance tests.
- TIF SQLite schema, migration, and project manager tests.
- GUI smoke tests for old JSON migration and SQLite project opening.
- TIF workbench tests for annotation tools, save state, undo/redo, local-axis review, and read-only behavior.
- Focused `py_compile` checks for changed GUI modules.
- `git diff --check` whitespace validation.

The branch has also been manually checked by the project maintainer for the 2D and TIF paths before this release.

## Data Boundary

This release does not include private research data.

The repository intentionally excludes:

- local project JSON files
- SQLite project databases
- TIF/CT stacks
- generated part volumes and reslices
- model weights
- runtime API settings
- private design notes under `private_docs/`

## Known Limitations

- TIF/CT remains a developer preview, validated mainly with AntScan ant CT data.
- TIF annotation does not yet include production lasso fill, bucket fill, or cross-slice propagation.
- Fully integrated TIF local-axis model training/inference is not yet a daily UI workflow; current emphasis is review, export, and training-material capture.
- Users should open the project entry file rather than the database file directly, though the app now tries to recover from that common mistake.
